# Render Library Deferred 3D Questions

Status: design questionnaire deferred beyond the first `ponder_render` milestone.

These questions cover the first 3D API and other rendering capabilities intentionally excluded from
the create/clear/present/resize/minimize/shutdown milestone, including scenes, resources, cameras,
picking, offscreen work, extensibility, later test tiers, and future Metal parity.

The fixed inputs in `open-questions.txt` continue to apply. Substantive project-owner answers are
preserved. Procedural answers that requested this move have been consumed and reset to a blank
`Answer:` line so the deferred design decision remains visibly open.

## Resolved Constraints Split From Bootstrap Questions

These later-only clauses came from mixed foundational questions whose bootstrap portions remain in
`open-questions.txt`. They are already accepted inputs, not questions being reopened:

- A device outlives its future resources, pipelines, scenes, and pending readbacks. Scenes reference
  resources without silently extending ownership; readbacks own their result state.
- An explicit synchronous readback may block. Other future operations should avoid hidden unbounded
  waits and expose intentional asynchronous completion.
- Reserve stable render error codes for shader, pipeline, upload, and readback failures when those
  APIs are introduced.
- Pending upload/readback and unavailable optional features are typed statuses unless the caller
  marked the capability or operation as required.
- Shader artifacts are generated for clean builds rather than controlled by a stale-artifact
  option. Add a Metal build option only when the native Metal backend exists.

## Initial Outcomes And Scope -- Resolve Before Choosing The API Shape

- **SCOPE-02:** What is the first real visualization vertical slice after bootstrap: atoms and
  bonds, generic meshes, points/lines/glyphs, molecular surfaces, volumetric data, or something
  else?

Recommendation: Use an interactive atoms-and-bonds scene as the first vertical slice, implemented
through reusable instanced glyph primitives. It exercises cameras, depth, dynamic attributes,
picking identity, and scientific scale while remaining representative of the product.

Answer:

- **SCOPE-03:** Which representations must the first 3D milestone selected in SCOPE-02 support, and
  which are explicit deferrals: ball-and-stick, space-filling, wireframe, ribbons, surfaces,
  scalar/vector/tensor fields, volumes, anisotropic glyphs, streamlines, uncertainty displays,
  labels, measurements, unit cells, and trajectories?

Recommendation: Include ball-and-stick, space-filling, wireframe, unit-cell lines, selection, and
per-frame trajectory coordinates in the first 3D milestone. Defer ribbons, generated surfaces,
volumes, field visualization, advanced glyphs, labels, and measurements until the generic scene and
resource-update contracts are proven.

Answer:

- **SCOPE-04:** Is the initial product strictly an interactive windowed desktop renderer, or must
  the first architecture also support headless devices, offscreen targets, readback, screenshots,
  image export, or remote execution?

Recommendation: Design device and render-target ownership for windowed and offscreen use from the
start, while implementing window presentation first and basic offscreen readback next. Scientific
automation, CI, and publication output make headless rendering a structural requirement even if it
is not in the bootstrap milestone.

Answer: Use the recommendation exactly

- **SCOPE-06:** Must a single scene and its GPU resources be renderable into multiple windows,
  multiple viewports, and offscreen targets without duplication?

Recommendation: Yes. Separate scene/resources from views and targets so immutable GPU data can be
shared while each target owns only size-, format-, and presentation-dependent state; this is
essential for comparison views and publication rendering.

Answer: Use the recommendation exactly

- **SCOPE-07:** What representative upper bounds should guide the design: atoms, bonds, instances,
  mesh triangles, volume dimensions, simultaneous scenes, trajectory frames per second, viewport
  resolution, and number of views?

Recommendation: Define benchmark tiers instead of public hard limits: 100,000 atoms plus bonds with
60 Hz coordinate streaming at 1440p on a pinned integrated-GPU system, and 1,000,000 at 30 Hz on a
pinned midrange discrete-GPU system. Add separate 10-million-triangle/4K and volume tiers when those
features enter scope.

Answer:

- **SCOPE-08:** What interactive performance and latency targets define success at those scales,
  and on what reference hardware classes?

Recommendation: Pin exact integrated and discrete adapter models, driver branches, OS builds, and
scene fixtures for the SCOPE-07 tiers. Require p95 frame times below 16.7 ms and 33.3 ms
respectively, visible interaction within two presented frames, and separately recorded submission,
GPU, upload-bandwidth, and presentation latency.

Answer:

- **SCOPE-09:** What image-quality requirements are non-negotiable for the first scientific scene,
  and which may be quality/performance settings?

Recommendation: Require correct geometry, depth, identity, linear color handling, stable camera
mapping, and readable edges. Make antialiasing level, impostor tessellation, shadows, transparency
quality, and post-processing explicit presets so performance choices never silently change
scientific meaning.

Answer:

- **SCOPE-12:** Which expected update pattern should dominate the first design: mostly static
  topology, per-frame coordinate changes for trajectories, frequent topology replacement,
  streaming surfaces/volumes, or a mixture?

Recommendation: Optimize first for stable topology with per-frame position and attribute updates,
plus efficient sparse changes and an explicit bulk-replacement path. That matches trajectories and
selection while leaving surface and volume streaming to later specialized resource paths.

Answer:

## Public Architecture And Library Boundary -- Foundational

- **ARCH-01:** Should the public API be a high-level visualization/scene API, a low-level
  backend-neutral GPU API, or layered with a public high-level API over a private low-level layer?

Recommendation: Expose a high-level domain-neutral scene/resource API over a private
backend-neutral GPU layer. Publishing a low-level graphics API would greatly expand the permanent
compatibility contract without helping the first scientific milestones.

Answer: Use the recommendation exactly

- **ARCH-02:** Should scene submission be retained mode, immutable per-frame snapshots,
  immediate-mode draw packets, or a deliberate hybrid?

Recommendation: Use a hybrid: retain long-lived resources, topology, and scene identity; apply
explicit batched updates; and submit immutable per-frame view/state snapshots. Reserve immediate
packets for transient overlays and debug drawing.

Answer:

- **ARCH-05:** May a process create multiple render devices or renderer instances? If so, may they
  share public resources or scenes?

Recommendation: Permit multiple devices for explicit adapter selection, isolation, and testing.
GPU resources and compiled scenes remain device-affine; reusable CPU descriptions may be compiled
independently for each device rather than implicitly shared.

Answer: Use the recommendation exactly

- **ARCH-07:** Should render publicly depend on `ponder::math` and accept its vectors, matrices,
  rays, frusta, and viewports, or use render-owned interchange values with explicit conversions?

Recommendation: Publicly depend on `ponder::math` and add narrow double-backed
`WorldPosition`/`WorldOrigin` values there as a prerequisite for the large-scene contract,
updating the math roadmap/ADR. Render should not invent a duplicate high-precision vocabulary.

Answer: Use the recommendation exactly

- **ARCH-08:** If offscreen support is required, should a platform-independent internal sub-target
  sit beneath `ponder_render` while the public render target retains its accepted platform
  dependency?

Recommendation: Put device-independent resource, scene, and frame logic in a private
platform-independent core beneath `ponder_render`, while retaining the accepted public platform
dependency for presentation. Headless use should not require a second public API.

Answer: Use the recommendation exactly

- **ARCH-13:** Should resource and scene references be move-only owners, opaque typed handles,
  generation-checked handles into renderer-owned storage, or another ownership model?

Recommendation: Represent each public independently owned resource as a move-only RAII wrapper
backed by an opaque generation-checked registry handle. Expose separate copyable typed references
for non-owning use in scenes/packets so ownership and stale-reference behavior are unambiguous.

Answer: Use the recommendation exactly

- **ARCH-14:** Is custom rendering intentionally closed to built-in features, or must application
  code and future plugins eventually add materials, shaders, primitives, or passes? 

Recommendation: Keep custom rendering closed to built-in features initially, but maintain a
private pass/resource architecture that can later support constrained extensions. Do not expose
raw shaders or arbitrary command access before enforcing lifetime and synchronization rules.

Answer: Use the recommendation exactly

- **ARCH-15:** If plugins will render, should they submit only stable, project-owned visualization
  data through the plugin SDK, or will a render-specific extension contract eventually be needed?

Recommendation: Plugins should initially submit versioned project-owned visualization data through
the plugin SDK. Add a separate render-extension contract only when real plugin requirements exceed
that model, with independent capability negotiation and versioning.

Answer: Use the recommendation exactly

- **ARCH-16:** Does a domain-neutral retained scene/visualization layer belong inside
  `ponder_render`, or in a separate future library that compiles scene data into a narrower render
  submission API? 

Recommendation: Keep a thin retained scene in render for generic primitives, resources, identity,
and updates. Put chemistry representations and higher-level visualization policy in a separate
layer that compiles into this narrow scene contract.

Answer: Use the recommendation exactly

## Lifetime, Threading, And Frame Ownership -- Foundational

- **LIFE-03:** Which operations may run concurrently: resource creation, CPU data preparation,
  uploads, shader/pipeline preparation, scene updates, command recording, readback, and queries?

Recommendation: Allow immutable descriptor, scene-packet, and source-data preparation on any
thread. Initially confine device mutation and submission to the render thread, adding concurrency
only through explicitly thread-safe upload/readback queues.

Answer: Use the recommendation exactly

- **LIFE-08:** Is ordinary GPU-resource destruction immediately blocking, explicitly deferred by
  a completion token, or automatically retired after in-flight frames finish?

Recommendation: Make ordinary destruction nonblocking and automatically retire backend objects
after referencing frames complete. Offer completion tokens only when callers must observe
reclamation or safely reuse external memory.

Answer: Use the recommendation exactly

- **LIFE-10:** When callers submit spans, strings, shader bytes, pixels, meshes, or instance data,
  are they copied before the call returns, borrowed through frame completion, or governed by an
  explicit upload object?

Recommendation: Copy descriptors, strings, and small payloads before returning. Large/asynchronous
data should use an explicit upload object that owns its bytes or returns a token defining the borrow
lifetime; never infer a long borrow from an ordinary span.

Answer: Use the recommendation exactly

- **LIFE-12:** Are asynchronous operations represented by polling handles, callbacks, futures,
  frame-completion tokens, or deliberately absent from the first API?

Recommendation: Use move-only polling handles/tokens with `IsReady`, bounded wait, and result
retrieval for uploads, readbacks, and completion. Avoid callbacks and `std::future` initially
because they impose scheduler, threading, and exception policy outside render's responsibility.

Answer: Use the recommendation exactly

## Device, Capability, Frame, And Presentation Policy -- Foundational

- **FRAME-16:** Should reverse Z or forward Z be the renderer default, and are both required per
  view/pass in the first scene API?

Recommendation: Default 3D views to reverse Z for materially better precision, while supporting
explicit forward Z per view for compatibility and tests. Projection, clear, comparison, readback,
and picking must follow the selected convention consistently.

Answer: Use the recommendation exactly

## Errors, Diagnostics, And Observability -- Foundational

- **ERR-04:** Apart from the device/surface-loss policy in FRAME-17, what is the policy for GPU
  out-of-memory, shader/pipeline failure, and partial resource-upload failure: retry, rollback,
  explicit renderer recreation, or unrecoverable termination?

Recommendation: Make resource, shader, pipeline, and upload operations transactional: roll back new
state, return a specific error, and preserve prior usable state. Allow explicit retry after
releasing memory; never publish a partially uploaded resource as successful.

Answer: Use the recommendation exactly

- **ERR-08:** Which public observations are required: selected adapter, enabled capabilities,
  present mode, memory budget/usage, resource counts, CPU/GPU timings, and frame statistics?

Recommendation: Always expose immutable selected-adapter, enabled-capability, and actual
presentation records. Provide budgets, resource counts, timings, and frame statistics through an
optional snapshot that explicitly marks unsupported measurements.

Answer: Use the recommendation exactly

- **ERR-09:** Is a backend-neutral frame capture/replay description required for debugging or
  reproducibility, or are external GPU capture tools sufficient?

Recommendation: Defer a persistent backend-neutral replay format; initially use external capture
tools, deterministic scene fixtures, and structured submission diagnostics. Keep packets
deterministic, but do not freeze a replay schema before scene/resource APIs stabilize.

Answer: Use the recommendation exactly

## Scene And Scientific-Visualization Boundary -- Before The First 3D API

- **SCENE-01:** Which library translates chemistry-domain objects into render data, given that
  `ponder_render` must remain chemistry-independent?

Recommendation: Put chemistry-to-render translation in a separate chemistry-aware visualization
layer, not `ponder_render`. That layer should emit backend-neutral scene data so chemistry rules
can evolve independently of GPU implementation details.

Answer:

- **SCENE-02:** What generic primitives should render understand directly: meshes, lines,
  polylines, points, sprites, glyph instances, analytic spheres/cylinders, volumes, text quads, and
  overlays?

Recommendation: Initially support indexed meshes, instanced analytic spheres/cylinders/glyphs,
points, lines/polylines, image or glyph quads, and overlay batches. Add scalar volumes as a
first-class generic primitive only when the later volume milestone can define and implement its
accuracy contract.

Answer:

- **SCENE-03:** Should chemistry-facing code submit semantic sphere/cylinder/glyph instances or
  always provide pre-tessellated generic geometry?

Recommendation: Make generic sphere, cylinder, and glyph instances first-class submissions while
retaining meshes as the general fallback. Semantic instances enable scalable batching, bounded LOD,
accurate radii, and backend optimization without forcing callers to tessellate.

Answer:

- **SCENE-04:** Who owns representation rules such as element radii/colors, bond-order layout,
  aromatic styling, orbital semantics, periodic replication, trajectory interpolation, and
  measurement formatting?

Recommendation: Put representation rules in the chemistry-aware visualization layer; render should
own only generic geometry, material, transfer-function, and composition behavior. This keeps
chemical meaning out of the backend-neutral rendering contract.

Answer:

- **SCENE-05:** How are immutable topology and frequently changing attributes such as positions,
  colors, radii, visibility, and selection state separated?

Recommendation: Separate stable topology from independently versioned dynamic attribute streams
for positions, colors, radii, visibility, and selection. Trajectories and interaction can then
update without rebuilding connectivity or unrelated GPU resources.

Answer:

- **SCENE-06:** What partial-update granularity is required, and must sparse updates be efficient?

Recommendation: Support contiguous range replacement and sparse index/value patches, applied
atomically at a defined frame boundary. Dense updates serve trajectories efficiently, while sparse
updates prevent selection or visibility changes from becoming full-buffer uploads.

Answer:

- **SCENE-07:** Must trajectory coordinates update every frame without reallocating topology or
  rebuilding unrelated GPU data?

Recommendation: Yes. Persistent topology and resource identities should allow coordinate and other
dynamic-stream changes every frame without reallocating unrelated data; staging, buffering, and
hazard management should remain private renderer responsibilities.

Answer:

- **SCENE-08:** Should scenes be mutable renderer-owned objects, immutable snapshots replaced as a
  unit, or caller-owned data compiled into explicit GPU resources?

Recommendation: Use retained renderer-owned compiled scenes addressed by typed handles, while
callers retain authoritative CPU data and submit immutable versioned snapshots or deltas. This
combines efficient GPU residency with deterministic updates and clear ownership.

Answer:

- **SCENE-09:** How does render carry caller/domain identity without depending on chemistry types:
  opaque 64-bit tokens, UUID-derived values, typed render IDs, or caller-maintained lookup tables?

Recommendation: Use strongly typed opaque 64-bit object and instance IDs with no domain semantics,
and let callers maintain domain lookup. Keep source-object and occurrence/replica identity distinct
so picking and updates remain unambiguous.

Answer:

- **SCENE-10:** Who creates spatial acceleration structures and owns their rebuild/refit policy:
  render, the chemistry/visualization layer, or a separate scene library?

Recommendation: Render should own acceleration structures needed for conservative culling,
rendering, and visible picking of its compiled scene; the visualization layer should own
authoritative source-data queries. Each can then rebuild or refit under its own correctness
contract without exposing backend structures.

Answer:

- **SCENE-11:** Are culling, batching, instancing, and level-of-detail decisions renderer-owned, or
  can callers provide pre-grouped/pre-culled submissions?

Recommendation: Render should own final view-dependent culling, batching, instancing, and LOD,
while callers provide semantic visibility filters, bounds, grouping hints, and permitted
representations. Pre-culled submissions would be wrong for multiple views and obstruct backend
optimization.

Answer:

- **SCENE-12:** Are molecular surfaces and isosurfaces generated outside render and submitted as
  meshes, or is any GPU surface extraction a render responsibility?

Recommendation: Generate molecular surfaces and isosurfaces outside render and submit them as
generic meshes initially. Future GPU extraction belongs in visualization or compute code that
produces the same mesh contract because scientific extraction is not rendering policy.

Answer:

- **SCENE-13:** Are volume textures and volume ray marching first-class render concepts? If so,
  what interpolation, sampling, and accuracy contract is required?

Recommendation: Add scalar volumes after the mesh milestone with explicit formats, transforms,
missing-value handling, interpolation space, transfer/compositing rules, gradient treatment, and
caller-selected step size. Classify the first ray marcher as deterministic qualitative
visualization with no convergence claim; add a bounded-error publication mode only when validated.

Answer:

- **SCENE-14:** Who owns scalar normalization, transfer functions, colormaps, missing-value colors,
  logarithmic mappings, and perceptual/color-vision-accessibility policy?

Recommendation: The visualization layer should explicitly choose normalization, ranges, mappings,
transfer functions, colormaps, and missing-value policy; render should apply them deterministically
in a declared color space. Render must never infer or silently rescale scientific data.

Answer:

- **SCENE-15:** Are annotations, atom labels, measurement text, legends, and axes scene primitives,
  UI overlays, or a separate overlay layer composed by render?

Recommendation: Treat world-anchored annotations and axes as scene content, and screen-space
labels, legends, and readouts as a backend-neutral overlay layer composed by render. Visualization
and UI own text, layout, and meaning; render owns only primitive composition.

Answer:

- **SCENE-16:** Which clipping and slicing features are required, and must cuts produce caps,
  contours, quantitative coordinates, or merely discard fragments?

Recommendation: Initially support explicit clipping planes, boxes, and slice planes with
deterministic fragment clipping. Quantitative contours, caps, and cut geometry should be produced by
the visualization layer because they require source-data-aware accuracy.

Answer:

- **SCENE-17:** What should happen when submitted positions, radii, scalar values, transforms, or
  colors contain NaN, infinity, missing, masked, or out-of-range values: reject the update, skip
  affected items, substitute diagnostics, or report indexed errors?

Recommendation: Reject non-finite geometry, transforms, radii, unmasked scalar values, and color
components with indexed/ranged diagnostics; never substitute silently. Require explicit missing
masks, and make out-of-range reject/clamp/map behavior a property of each format and transfer
function.

Answer:

- **SCENE-18:** Is render entirely unit-agnostic? If so, what finite coordinate/radius range must
  callers obey, and where are conversions from chemistry units performed?

Recommendation: Keep render unit-agnostic with one caller-defined world unit per scene; chemistry
conversion stays in visualization code. Use math-owned `WorldPosition`/`WorldOrigin` values with
local FP32 chunk positions and explicit precision bounds for large scenes.

Answer:

- **SCENE-19:** Must periodic replicas preserve both source-object identity and replica identity for
  picking, selection, and updates?

Recommendation: Yes. Carry stable source identity separately from replica/instance identity so
shared source updates remain efficient while selection, picking, and measurement of individual
periodic images stay deterministic.

Answer:

- **SCENE-20:** May LOD/impostor substitutions alter silhouettes or apparent radii, and must
  quantitative/picking behavior still use exact intended geometry?

Recommendation: Permit presentation LOD only within an explicit caller-selected screen-space error
bound, without silently changing intended quantitative dimensions. Picking and measurements should
resolve canonical geometry unless a query explicitly requests the displayed approximation.

Answer:

## Camera, Coordinates, Precision, And View State -- Before The First 3D API

- **VIEW-01:** Does render own a public camera type, or does the caller submit `ponder_math` view
  and projection matrices plus explicit depth convention each frame?

Recommendation: Render should own a lean backend-neutral `View` descriptor, not navigation or
camera behavior. It should consume `ponder_math` transforms or canonical projection parameters
with explicit depth, viewport, and origin information, then normalize backend differences.

Answer: Use the recommendation exactly

- **VIEW-02:** Must one target support multiple cameras, sub-viewports, scissors, stereo views, or
  layered views initially or only remain extensible to them?

Recommendation: Shape target submission around a list of views with independent viewports and
scissors from the beginning, even if the first implementation permits one. Stereo and layered
rendering can remain explicit later capabilities without assuming one camera per target.

Answer: Use the recommendation exactly

- **VIEW-03:** At what render boundary should the required Vulkan viewport-Y adaptation occur, and
  what front-face/winding rule must shaders, meshes, culling, and the future Metal backend observe?

Recommendation: Perform viewport-Y and winding adaptation only at the private backend boundary.
Public geometry and shaders should consistently use project coordinates and counter-clockwise front
faces, with each backend adapting viewport/projection and culling together.

Answer: Use the recommendation exactly

- **VIEW-04:** What are the canonical shader matrix multiplication, storage, transpose, and GPU
  packing rules for `ponder_math` column-major values?

Recommendation: Define shader math as column-vector multiplication,
`clip = projection * view * model * position`, with conceptually column-major matrices and no
caller transposition. Render should own explicit GPU packing generated or validated by reflection.

Answer: Use the recommendation exactly

- **VIEW-05:** Given the float-only math contract, what strategy supports large molecules,
  periodic systems, and widely separated structures: documented float limits, camera-relative
  coordinates, origin rebasing, spatial partitioning, double input converted at submission, or a
  combination?

Recommendation: Store geometry as FP32 positions local to spatial chunks with the math-owned
double-backed `WorldOrigin`, deriving per-view origin transforms without duplicating geometry.
Use deterministic rebasing and expose conversion error bounds so simultaneous views share resources
without silently losing scientific scale.

Answer:

- **VIEW-06:** Who selects near/far planes, and must infinite-far projection be supported by the
  standard camera path?

Recommendation: Require callers to supply explicit near/far policy or opt into validated
bounds-based fitting; render should not silently guess. Support reverse-Z infinite-far perspective
as the preferred standard path while retaining explicit finite and forward-Z options.

Answer: Use the recommendation exactly

- **VIEW-07:** Are atom, bond, glyph, line, and point sizes always world-space, screen-space, or
  selectable? How are quantitative and presentation-oriented modes distinguished?

Recommendation: Make size space explicit per primitive: world-space by default for quantitative
atoms, bonds, and glyphs, and screen-space for aids such as thin lines, points, and labels. The mode
must also define picking and measurement semantics.

Answer:

- **VIEW-08:** How are platform logical coordinates, window pixel coordinates, render-target
  coordinates, viewport coordinates, and NDC converted under high pixel density?

Recommendation: Provide explicit conversions using a per-frame pixel-scale and viewport snapshot,
with typed coordinate values and documented origins, extents, pixel-center rules, and rounding.
This prevents high-density displays from making projection and picking ambiguous.

Answer: Use the recommendation exactly

- **VIEW-09:** Does render own scene-level frustum/occlusion culling, or should callers submit
  already culled objects while continuing to use `ponder_math` for project/unproject operations?

Recommendation: Render should own conservative view-dependent frustum, LOD, and optional occlusion
culling from caller-supplied bounds; callers provide semantic visibility filters or coarse
partitions. Keep project/unproject behavior in backend-neutral math/view utilities.

Answer: Use the recommendation exactly

- **VIEW-10:** Are temporal jitter, motion vectors, stereo transforms, and camera history deliberate
  future contracts, or should the first camera API avoid implying them?

Recommendation: Make the first view descriptor stateless per frame and avoid hidden camera
history. Reserve explicit optional history, previous transforms, jitter, and motion inputs for
temporal techniques so ordinary camera semantics remain simple.

Answer: Use the recommendation exactly

- **VIEW-11:** What numeric formats, quantization, clamping, and permitted error apply when source
  positions, radii, scalar attributes, meshes, or volumes are converted to GPU representations,
  especially when source data is double precision?

Recommendation: Require callers to declare source format, legal range, and required precision/error,
not a backend GPU layout. Render selects and reports a compliant internal format or fails; FP32
camera/chunk-relative geometry is the default, and quantization is opt-in.

Answer: Use the recommendation exactly

## Resource, Shader, Material, And Pass Design -- Before General Drawing

- **RES-01:** Which resource vocabulary is public at first: buffers, images, samplers, render
  targets, shader modules, pipelines, materials, meshes, and descriptor/binding objects?

Recommendation: Initially expose move-only high-level mesh, texture, material, render-target, and
scene owners backed by generation-checked handles, with separate typed non-owning references. Defer
volume resources to their milestone, and keep generic buffers, shaders, pipelines, descriptors, and
bindings private.

Answer:

- **RES-02:** Are resource creation and uploads synchronous initially, asynchronous with completion
  tokens, or internally staged while exposing a readiness state?

Recommendation: Design uploads as asynchronous operations with readiness/completion tokens, plus an
explicit blocking convenience for initialization and tests. This preserves scalability and makes
synchronization visible without exposing backend fences.

Answer: Use the recommendation exactly

- **RES-03:** May resources be updated in place while in flight, and how are update hazards exposed
  or hidden?

Recommendation: Present updates as versioned replacements that become visible atomically at a
defined frame boundary. Hide hazards through internal renaming, staging, and deferred retirement so
callers never manage barriers or mutate GPU-consumed bytes.

Answer: Use the recommendation exactly

- **RES-04:** Are CPU/GPU memory budgets queryable, enforced, or advisory? Is automatic eviction or
  out-of-core paging required for large meshes and volume bricks?

Recommendation: Expose budget/usage estimates, enforce optional caller-configured hard budgets, and
return actionable allocation failures. Automatically evict only resources explicitly declared
reloadable or streamable; silent arbitrary eviction would harm determinism.

Answer: Use the recommendation exactly

- **RES-05:** Which GPU packing layouts should render own for vertices, instance data, uniforms,
  storage data, and push constants? Are direct byte uploads from math values forbidden unless an
  explicit packing contract says otherwise?

Recommendation: Render should own every vertex, instance, uniform, storage, and constant packing
layout and validate it against shader reflection. Forbid direct byte uploads from ordinary math or
C++ objects unless the API names an explicit versioned packed layout.

Answer: Use the recommendation exactly

- **RES-06:** Should clients submit general command buffers/passes, typed draw lists, or only
  high-level scene objects?

Recommendation: Accept high-level scene objects and constrained typed draw packets, not general
command buffers or native passes. This keeps synchronization, state transitions, and cross-backend
correctness enforceable.

Answer: Use the recommendation exactly

- **RES-07:** Is a render/pass graph public, private, or deferred until multiple real passes require
  resource scheduling and transient allocation?

Recommendation: Keep the graph private and introduce it internally when real multipass scheduling
requires it. Public APIs should describe semantic outputs/effects; expose graph extension points
only after concrete needs define safe invariants.

Answer: Use the recommendation exactly

- **RES-08:** What shader source language/toolchain should serve Vulkan now and native Metal later:
  one portable source compiled to SPIR-V and Metal output, or backend-specific sources behind shared
  semantic interfaces?

Recommendation: Use one portable renderer-owned shader source and semantic interface for SPIR-V and
Metal, with isolated backend-specific code only where necessary. Before selecting the toolchain,
prototype and compare both output targets plus reflection/debugging; two functioning runtime
backends are not required for that prerequisite.

Answer:

- **RES-09:** Are shaders compiled offline at build time, during asset processing, or at runtime?
  Is developer hot reload required?

Recommendation: Compile production shaders offline during the build/asset pipeline and fail early
on compilation, validation, or interface mismatch. Developer hot reload may reuse that compiler
path, but runtime compilation should not be a deployed dependency.

Answer: Use the recommendation exactly

- **RES-10:** Are compiled shader artifacts checked in, generated in the build tree, or packaged as
  runtime assets? How are their versions tied to the renderer?

Recommendation: Generate artifacts in the build tree and embed them into the backend static library
instead of checking opaque binaries into source or locating runtime files. Generated metadata
should bind each artifact to source hashes, tool versions, backend, and renderer interface version.

Answer: Use the recommendation exactly

- **RES-11:** Is reflection required to generate/validate resource layouts, vertex formats, and
  pipeline interfaces, or are layouts declared manually in C++ and shaders?

Recommendation: Require reflection to generate or validate bindings, vertex inputs, constants, and
pipeline interfaces. One reflected contract avoids duplication between C++ and shaders and enables
cross-backend conformance tests.

Answer: Use the recommendation exactly

- **RES-12:** May applications or plugins supply shader code and custom materials, or are all
  shaders and pipelines renderer-owned?

Recommendation: Keep shaders and pipelines renderer-owned initially. If customization becomes
necessary, add a constrained versioned backend-neutral material/shader extension rather than
accepting arbitrary backend code through the core API.

Answer: Use the recommendation exactly

- **RES-13:** Who owns persistent pipeline-cache bytes and their storage policy, given render should
  not own general filesystem or application configuration behavior?

Recommendation: Maintain an in-memory cache and expose validated opaque cache blobs through
caller-provided load/save data. The application owns persistence location/policy; render owns
fingerprints, compatibility validation, and safe rejection.

Answer: Use the recommendation exactly

- **RES-14:** What is the first lighting/material contract: unlit data-faithful color, illustrative
  lighting, physically based rendering, or several explicitly named modes?

Recommendation: Make data-faithful unlit color mandatory and offer illustrative lighting as a
separate named mode for shape perception. Defer PBR until a real need appears, and never let
lighting become an implicit transformation of scalar-derived colors.

Answer:

- **RES-15:** What transparency correctness is required for overlapping atoms, bonds, surfaces, and
  volumes: sorted blending, an approximate order-independent method, exact order-independent
  transparency, or feature-specific policies?

approximation, and depth peeling for publication until measured residual transmittance falls below
a declared tolerance or a reported limit is hit. Never silently downgrade a requested guarantee.

Recommendation: Require a backend-neutral baseline using supported multisampling plus analytic
coverage for line/glyph edges, with deterministic supersampling for export. Make temporal
reconstruction optional and off by default because history can blur scientific detail.

## Picking, Selection, And Interaction -- Before Interactive Scene Features

- **PICK-01:** Is picking owned by render, by math/application code, or by a hybrid in which render
  supplies visible IDs/depth and another layer resolves domain objects?

Recommendation: Use a hybrid: render performs visibility/geometry-aware queries and returns
backend-neutral render IDs and hits; visualization/domain code resolves authoritative chemistry
objects. This preserves clipping/LOD knowledge without domain types in render.

Answer:

- **PICK-02:** Which entities must be pickable: atoms, bonds, glyphs, mesh objects/faces/vertices,
  annotations, periodic images, volume samples, and isosurfaces?

Recommendation: Every supported primitive should declare pickability and return object, instance,
and meaningful primitive identity. Initially cover atoms/glyphs, bonds, mesh objects/faces,
annotations, and periodic images; specialize vertex and volume sampling separately.

Answer:

- **PICK-03:** Should picking use CPU ray queries, a GPU ID/depth pass, hardware ray queries where
  available, or multiple strategies behind one semantic contract?

Recommendation: Provide one semantic query API backed by GPU ID/depth for displayed pixels, CPU
analytic/BVH queries for canonical geometry, and optional hardware ray queries as an optimization.
Callers should choose semantics, not a GPU mechanism.

Answer:

- **PICK-04:** Must a hit represent exact analytic geometry, submitted mesh geometry, or the pixels
  actually visible after LOD, clipping, transparency, and post-processing?

Recommendation: Expose distinct variants: a visible sample reads a pre-postprocess attribution
attachment and may report no contributor or multiple weighted contributors, while a canonical hit
reports exact geometry. Canonical refinement is an explicit follow-up using a contributor's source
identity.

Answer:

- **PICK-05:** Is selection visible-surface-only, or must callers request through-selection,
  occluded candidates, and all hits?

Recommendation: Default to the nearest visible surface, but support explicit through-selection and
all-hit modes with deterministic ordering. Scientific selection often requires occluded
structures, so visibility must be query policy.

Answer:

- **PICK-06:** What result data is required: caller identity, primitive kind, instance/replica ID,
  world hit point, normal, distance/depth, barycentrics, scalar value, and screen position?

Recommendation: Use tagged visible and canonical variants tied to a query snapshot. Visible results
carry pixel/depth plus zero or more weighted object/instance contributors; canonical hits carry one
identity, world point, normal, distance, and only meaningful barycentric/scalar fields.

Answer:

- **PICK-07:** Must point/hover picking be synchronous, asynchronously returned after GPU readback,
  or available in both forms with different guarantees?

Recommendation: Make asynchronous completion standard for GPU hover and batch queries, with an
explicit blocking convenience for clicks, CPU-resident queries, and tests. Both forms must use the
same result semantics and identify the queried scene/frame snapshot.

Answer:

- **PICK-08:** Are rectangle, lasso, multi-hit, and nearest-N queries initial requirements or
  explicit deferrals?

Recommendation: Include point-nearest, rectangle, and multi-hit/through selection in the initial
interaction contract. Defer lasso and nearest-N implementation behind an extensible query-shape
and cardinality model rather than hard-coding only point picking.

Answer:

- **PICK-09:** What deterministic tie-breaking applies to coincident or nearly coincident hits?

Recommendation: Build a transitive total-order key from a documented quantization of canonical
distance or visible depth, followed by stable object, instance, primitive-kind, and primitive IDs.
Require exact identity for canonical queries; test visible queries away from raster boundaries or
with explicitly tolerance-bounded attribution.

Answer:

- **PICK-10:** How should transparent surfaces and volumetric data participate in picking?

Recommendation: Canonical queries return ordered geometry intersections. Visible queries use the
pre-postprocess attribution buffer and may return weighted transparent contributors under a
declared cutoff; volume attribution uses a stated accumulated-opacity/scalar criterion, while
quantitative sampling remains separate.

Answer:

- **PICK-11:** Who owns selection visualization such as color changes, outlines, glow, halos, and
  hover state: render, the scene producer, or UI/overlay code?

Recommendation: Visualization owns selected/hovered state and styling intent; render provides
generic configurable tint, outline, halo, or mask composition. UI owns interaction timing, and
render should never choose chemistry-specific selection meaning.

Answer:

- **PICK-12:** Do tiny points and lines require a configurable screen-space picking aperture and a
  stable ranking rule when several candidates fall within it?

Recommendation: Yes. Resolve a per-query screen-space aperture using the frame pixel scale, rank by
screen-space distance, then depth and the stable ID tuple, and return the canonical geometry
location separately.

Answer:

- **PICK-13:** Must measurement tools resolve original domain coordinates rather than approximate
  LOD/impostor hit geometry, and which layer performs that resolution?

Recommendation: Yes. Render returns stable IDs and canonical hit information; visualization/domain
code maps those IDs to original-precision coordinates and performs measurements, never relying on
LOD, raster, depth-buffer, or impostor approximations.

Answer:

Recommendation: Return structured metadata covering renderer/shader versions, backend and

## Compute, Multi-GPU, And Extensibility -- May Be Deferred, But Shape Ownership

- **EXT-01:** Will compute-generated geometry, trajectories, and volumes initially cross into
  render through CPU memory, or is zero-copy GPU interoperability a requirement?

Recommendation: Start with explicit CPU uploads using reusable staging and partial updates. Preserve
an imported-resource extension point, but do not burden the stable API with zero-copy
synchronization before a concrete producer requires it.

Answer:

- **EXT-02:** If GPU interop is required, which ecosystems matter (Vulkan compute, CUDA, HIP, SYCL,
  OpenCL, or external processes), and which side owns device selection?

Recommendation: Require no external ecosystem initially; use the render device for render-owned
Vulkan compute, then add CUDA/HIP/other integrations only for concrete consumers. The application
coordinates device choice using stable adapter identity.

Answer:

- **EXT-03:** Who owns external-memory/semaphore import/export and cross-queue synchronization, and
  may those native mechanisms appear only in a separate backend-specific interop module?

Recommendation: Put native external-memory, semaphore, and queue ownership in a separate
backend-specific interop module. Render retains lifetime/state-transition authority through opaque
resources and completion tokens; native handles never enter the general API.

Answer:

- **EXT-04:** Must rendering and compute share one physical GPU, support peer transfer across GPUs,
  or remain independent with explicit staging?

Recommendation: Require the same physical GPU for zero-copy interoperability and use explicit CPU
staging when adapters differ. Peer transfer should be a later negotiated capability, not an
implicit fallback.

Answer:

- **EXT-05:** Is explicit multi-GPU rendering or adapter migration a requirement, a future
  capability to preserve, or out of scope?

Recommendation: Keep multi-GPU rendering and live migration out of scope while permitting
independent devices. Resources remain device-local; caller-owned source data is re-uploaded through
an explicit reconstruction path.

Answer:

- **EXT-06:** If custom passes become possible, what prevents extensions from violating resource
  lifetime, synchronization, color/depth, and deterministic-order contracts?

Recommendation: Expose typed declared inputs, outputs, usages, and a scoped renderer-controlled
encoder. Render owns scheduling, barriers, transient lifetimes, conventions, and ordering;
extensions may neither retain transient handles nor access native devices/queues.

- **EXT-07:** Does a future plugin-facing visualization contract need versioning/capability

Recommendation: Yes. Give the plugin-facing contract independent versioning and capability
negotiation; plugins exchange stable project-owned data through the SDK, and an adapter compiles it
into the current internal render submissions.

Recommendation: Require backend-free state/lifetime tests, public-header and shader-interface

Recommendation: Use a pinned lavapipe build as the required deterministic headless CI path, plus
