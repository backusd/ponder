# Render Library Roadmap

Status: implementation plan for the first real `ponder_render` milestone.

This roadmap implements and validates Vulkan
create/clear/present/resize/minimize/shutdown across multiple platform windows. Project-owned UI
drawing and the first 3D visualization are separate later milestones. Their design questions live in
`../../ui/docs/open-questions.txt` and `open-questions-3d.txt`.

The decisions in `open-questions.txt` are resolved inputs to this plan. REND-000 and REND-001 first
remove the legacy platform/render contracts that still describe macOS Vulkan portability.
Implementation must not begin on top of those contradictory contracts.

Every task must leave the repository configured, buildable, and tested for the behavior introduced
by that task. Every task that adds or changes a public header must add or update that header's
dedicated PCH-free self-containment translation unit in the same task. Tests accompany the
production behavior they protect rather than being deferred wholesale to the end.

## Guiding Decisions

- The milestone is complete only when multiple windows can be created, cleared, presented,
  resized, minimized/restored, independently destroyed, and shut down without validation errors.
- Vulkan is the build-selected backend on Windows and Linux. There is no runtime backend selector.
  `GetRequiredWindowGraphicsCompatibility() noexcept` remains parameterless.
- `ponder_render` does not build on macOS until a native Metal backend exists. Configuring render
  on macOS fails clearly; configuring the repository with render disabled remains supported.
- Vulkan 1.2 is the minimum device contract for integrated and discrete GPUs. Build against pinned
  current Vulkan headers, query loader and physical-device versions independently, and enable only
  functionality actually supported by the selected device.
- Vulkan 1.4 is not required. The bootstrap's required path needs no optional device feature bit
  and requires only `VK_KHR_surface`, the active Windows/X11/Wayland WSI extension, and
  `VK_KHR_swapchain`. Optional presentation-completion extensions improve retirement guarantees.
- The system Vulkan loader is opened dynamically through private Volk integration. A Vulkan SDK
  and an import-library link are not runtime or consumer requirements.
- Public headers expose only project-owned, standard-library, core, and platform types. SDL,
  Vulkan, Volk, VMA, native OS declarations, and backend targets remain private.
- `RenderDevice` lifetime is separate from window-target lifetime. One device serves independent
  targets for multiple windows; no process-global renderer singleton is introduced.
- The first device is selected with presentation requirements from an opaque prepared surface.
  Later targets are checked against that device and fail explicitly if its adapter cannot present
  to their surfaces. Render never migrates resources or silently creates another device.
- A borrowed native-window snapshot is acquired and consumed synchronously while creating its
  Vulkan surface on the platform owner thread. It is never stored or passed to another thread.
- Only the resulting render-owned opaque surface handoff and copied project-owned window-state
  values may cross to the caller-selected render thread. Vulkan device mutation and frame
  submission are affine to that thread, which may also be the platform owner. Render creates no
  hidden thread.
- Render never calls `Window::GetNativeHandle()` from a dedicated render thread, retains a
  long-lived `Window` borrow, or keys state by a `Window*`.
- The first owner-thread surface preparation selects the one host WSI instance extension. A
  platform runtime has one active SDL video driver, so later windows must use the same native
  variant; a mismatch fails explicitly.
- Device creation provisions the queue families needed for later presentation targets. The
  implementation supports separate graphics and presentation families and does not assume a
  combined family.
- A move-only `RenderBootstrap` owns loader/instance state and remains on the platform owner thread
  to prepare surfaces. A render-thread `RenderDevice` is its child and owns device-level resources;
  targets are device children.
- These owners use heap-stable RAII state and release-active child registries. Application
  shutdown destroys unconsumed prepared surfaces and targets before the device and bootstrap,
  then destroys platform windows and finally the platform runtime.
- A target owns its surface, swapchain, image views, clear-pass objects, per-frame synchronization,
  dirty/suspended state, and actual selected presentation configuration.
- Platform pixel size is authoritative. Logical size is not used for swapchain extent. Minimized,
  hidden, and zero-pixel targets return a successful typed suspended/skipped result and perform no
  acquire or present work.
- Swapchain out-of-date and suboptimal results trigger internal target-local recreation and are
  reported as metadata. Surface loss requires a fresh owner-thread window snapshot; device loss is
  fatal to that device.
- Presentation policy is semantic: `VSync` by default, `LowLatencyVSync`, and `Uncapped`. Required
  choices fail when unavailable; preferences return the actual fallback and reason.
- The default queued-frame-latency request is two, with a documented supported range of one through
  three. Raw image counts, Vulkan modes, and synchronization topology remain private.
- The initial output is opaque SDR sRGB with linear-light clear input and explicit encoding to the
  selected sRGB presentation format. HDR, wide gamut, offscreen alpha, and export are deferred.
- A move-only frame token makes call order explicit. Only one mutable frame exists per target;
  accepted submissions have deterministic order; abandoned unfinished tokens retire safely.
- Public recoverable failures use `Result` and stable render error codes in
  `0x0003'0000` through `0x0003'FFFF`. Native result values are optional diagnostic context, never
  public Vulkan types or control-flow codes.
- Validation warnings and errors fail validation tests. Filters are exact message-ID and
  vendor/driver scoped, justified, owned, and time-bounded; informational messages do not fail.
- UI drawing is not implemented here. The finished frame/target internals must leave a documented
  generic 2D insertion point so a project-owned UI paint compiler can submit owning packets without
  replacing device, swapchain, or frame ownership.
- Windows on the current development laptop is the only initially verified runtime host. Linux X11
  and Wayland remain implemented and compile-covered but explicitly unverified until exercised on
  native hosts.

## Scope Exclusions

This roadmap does not implement scenes, cameras, public GPU resources, mesh or texture uploads,
shaders, materials, picking, depth policy, offscreen targets, readback, image comparison, software
ICD qualification, compute interop, plugins, multi-GPU, runtime backend choice, Metal, or Dear
ImGui. Internal clear-pass Vulkan objects are implementation details, not a general drawing API.

## Milestones

| Milestone | Tasks | Result |
| --- | --- | --- |
| Contracts | REND-000 through REND-001 | Agreed native Vulkan and Metal boundaries |
| Build foundation | REND-002 through REND-004 | Conditional targets and bootstrap API |
| Vulkan bootstrap | REND-005 through REND-010 | Handoff, instance, surfaces, and device |
| Presentation | REND-011 through REND-016 | Clear, recovery, and multiple windows |
| Handoff | REND-017 through REND-020 | Tests, example, and intermediate-draw-ready gate |

## Verification Cadence

Use one configured supported Debug developer tree for intermediate tasks. Reconfigure only when
CMake inputs, dependency wiring, or source lists change.

- Public header or value contract: build `ponder_render_header_tests` and
  `ponder_render_tests`; run the complete value test executable.
- Private loader, WSI, device, swapchain, or frame behavior: build and run
  `ponder_render_backend_tests`.
- Both public and private behavior: build all three targets above and run both deterministic test
  executables.
- Applicable real Vulkan/window behavior: also build `ponder_render_integration_tests` and run its
  live tests through one CTest invocation.
- Platform interop contract: build affected platform header, value, backend, and integration
  targets before render tests.
- Example source: build `ponder_render_clear_window`; do not register the interactive example with
  CTest.
- Documentation only: do not rebuild unless CMake, commands, generated interfaces, or public
  contracts changed.

Live render integration tests use the same `ponder_platform_sdl` CTest resource lock as platform
integration tests because both own process-global SDL runtime state. A supported-environment job
fails when its required loader, validation layer, adapter, or WSI capability is missing. An
explicitly optional local or compile-coverage job may skip only after identifying the precise
missing capability; an unexpected initialization failure is not a skip.

TASK REND-005: Define Threading, Lifetime, And Window-State Handoff
-------------------------------------------------------------------

Status: completed on 2026-07-13.

Define the two-thread contract without extending the lifetime of borrowed native data.

Work:

- Add an immutable target-state snapshot containing stable `WindowId`, pixel size, visibility,
  faithful platform `WindowState`, and a typed presentation-environment revision. It contains no
  native handle or borrowed pointer.
- Define `RenderBootstrap` as the heap-stable owner-thread root for loader/instance state and
  surface preparation. It must outlive every prepared surface, device, and target created from it.
- Give the bootstrap a synchronized release-active child registry for prepared surfaces and
  devices so teardown cannot race or silently orphan instance children.
- Require the application to coalesce platform notifications and capture the latest state on the
  platform owner thread before handing that copied value to render. That application-side
  aggregator owns the monotonic per-window snapshot and presentation-environment revisions.
- Define a move-only opaque prepared-surface handoff whose public representation contains no
  Vulkan, native-window, SDL, or operating-system declaration.
- Specify that only an owner-thread surface-preparation operation may accept a platform `Window`.
  That operation must acquire and consume `NativeWindowHandle` synchronously in the same call.
- Forbid storing, returning, or passing the borrowed native snapshot to another thread, even behind
  an opaque wrapper. The later handoff owns a completed render surface, not native source data.
- Permit the completed opaque surface and copied target state to transfer once to the
  caller-selected render-affine thread, accepting the identical preparation snapshot or a genuinely
  newer monotonic snapshot. Device mutation and submission begin there while the bootstrap remains
  on the platform owner thread.
- Copy every descriptor and snapshot needed after the call. Do not retain `Window&`, `Window*`, or
  references into caller storage.
- Define a fresh owner-thread surface-preparation call for surface-loss recovery. Ordinary
  swapchain resize uses only target state and never reacquires the native handle.
- Define moved-from, wrong-window, stale-state, wrong-thread, and shutdown behavior before Vulkan
  implementation begins.
- Add deterministic tests for descriptor copying, invalid compatibility, malformed values, stale
  state ordering, move stability, and thread-affinity checks that require no Vulkan loader.

Done when:

- No valid public state can carry a borrowed native payload beyond its owner-thread call, and a
  dedicated render thread never needs to call a platform `Window` method.
- Target state is routed by `WindowId`, and moving either the platform window owner or later render
  owners cannot invalidate copied public state.

TASK REND-006: Implement Dynamic Loading And Instance Bootstrap
---------------------------------------------------------------

Status: completed on 2026-07-13.

Establish transactional Vulkan process/instance state behind a narrow dispatch seam.

Work:

- Open the host's standard Vulkan loader dynamically through Volk when Vulkan work is first
  requested. Return `LoaderUnavailable` with actionable context if it cannot be opened.
- Query instance support with `vkEnumerateInstanceVersion` and require at least Vulkan 1.2.
  Negotiate the highest version understood by the pinned headers and loader, while never using a
  device command or feature above the selected physical device's version.
- Let the owner-thread surface-preparation path supply an internal Win32, X11, or Wayland WSI kind
  before instance creation. Enable exactly `VK_KHR_surface` and that kind's instance extension.
- Store the selected WSI kind in private instance state and reject later preparation requests for a
  different kind.
- Use private move-only RAII wrappers and an instance dispatch table rather than exposing Vulkan
  ownership or relying on exception-driven wrappers.
- Store instance state in the heap-stable `RenderBootstrap`. Serialize its owner-thread surface
  operations against instance teardown while allowing device operations on the render thread.
- Add a narrow injectable loader/instance dispatch seam. Test missing loader, loader below 1.2,
  missing required extensions, instance creation failure, repeated initialization, and rollback.

Done when:

- Private instance creation and destruction succeed without a window surface, every partial failure
  unwinds cleanly, and the selected API version and WSI contract are queryable privately.
- No loader is opened during CMake configuration or merely by linking a consumer.

TASK REND-008: Create Owner-Thread Vulkan Surfaces

Status: completed on 2026-07-13.

Consume the borrowed platform native handle entirely inside its valid owner-thread call.

Work:

- Implement `RenderBootstrap`'s public owner-thread preparation operation defined in REND-005. Its
  first call
  establishes the exact-WSI instance bootstrap and surface atomically; later calls reuse that
  instance through the same bootstrap.
- Call `Window::GetNativeHandle()` and map `NativeWin32Window`, `NativeX11Window`, or

  Never include SDL or call `SDL_Vulkan_CreateSurface`.

- Return a move-only opaque prepared-surface value that owns the resulting `VkSurfaceKHR`, stable

- Permit that completed value to move to the render thread. Surface destruction happens only after
  render releases all use and never requires another platform native-handle query.
- Reuse the same operation for a replacement after surface loss; ordinary swapchain recreation

- Test each native alternative and injected invalid, wrong-WSI, instance, native-call, allocation,

- Add a live Windows owner-thread surface create/destroy test using a platform window.

Done when:

- Native payload inspection and `vkCreate*SurfaceKHR` occur synchronously on the platform owner
  thread, while the completed opaque surface can safely transfer to the render thread.

TASK REND-012: Implement Swapchain Negotiation And Creation
-----------------------------------------------------------

Status: completed on 2026-07-14 after the final-audit corrections recorded below.

Add presentation resources to the tested target owner.

Work:

- Query surface capabilities, formats, and present modes on every creation or recreation. Treat
  capabilities as live data rather than adapter constants, and repeat format and present-mode
  count/read enumeration when Vulkan reports `VK_INCOMPLETE` instead of accepting a partial list.
- Require `VK_FORMAT_B8G8R8A8_SRGB` or `VK_FORMAT_R8G8B8A8_SRGB` with
  `VK_COLOR_SPACE_SRGB_NONLINEAR_KHR` and opaque composition; prefer BGRA deterministically when
  both formats are present. Reject undefined, UNORM, other color spaces, and non-opaque-only
  surfaces with `UnsupportedSurface`. Record the exact selected format, color space, alpha mode,
  pixel extent, semantic presentation policy, and semantic queued latency in the project-owned
  selected-configuration snapshot.
- Map `VSync` to FIFO. Map `LowLatencyVSync` to mailbox with FIFO fallback. Map `Uncapped` to
  immediate with mailbox then FIFO fallback. Fail an unavailable requirement; for a preference,
  return the selected fallback and reason.
- Interpret queued-frame latency as a maximum number of accepted frames still outstanding, with a
  supported request range of one through three. Enforce it through per-target frame
  synchronization/acquisition scheduling and expose the actual semantic limit.
- Select the private swapchain image count independently from surface limits and present-mode
  requirements. A surface-mandated larger image count must not raise the semantic latency; limited
  image availability may lower the achievable latency and must be reported.
- Support exclusive sharing when graphics and presentation use the same family and correct
  concurrent or ownership-transfer behavior when they differ. Keep this choice private.

## Final Audit

Audit date: 2026-07-14.

This audit covers the resolved decisions in `open-questions.txt`, the work described by REND-000
through REND-018, the current public and private render implementation, and every render unit,
header, consumer, install-consumer, and live integration test. It is a design and source audit, not
an assertion that every current test failure would expose the issues below. In several important
cases, the current tests encode the incomplete behavior or never reach the affected production
path.

### Outcome At Audit Time

and two-window happy paths are useful foundations, but REND-011 through REND-018 are only partially
complete. There are correctness gaps in presentation synchronization, swapchain retirement and
recreation, lifetime enforcement, and loss recovery. There are also public-contract mismatches and
verification gates that currently report more confidence than the tests provide. Publishing the
minimal example now would make those incomplete choices the recommended application pattern.

The strongest existing work should be retained:

- Stable render error values, public enum/value validation, and move-only API traits have direct
  tests.
- Both public headers have PCH-free self-containment translation units, and the build-tree consumer
  uses explicit public targets with strict warnings.
- Fake dispatch gives useful deterministic coverage for loader/instance creation, validation-mode
  selection, WSI routing, adapter ranking, queue-family selection, device creation, and basic
  swapchain selection.
- The Windows live tests create real Vulkan surfaces, devices, swapchains, and frames, and they
  demonstrate basic two-window independence with finite CTest timeouts and the shared SDL resource
  lock.

### Follow-Up Disposition (2026-07-14)

Every correctness blocker, public-contract correction, and test-audit finding below now has an
implemented resolution, and the UI-000 source audit reconfirmed those invariants. REND-019 has
therefore been implemented. This supersedes the historical instruction not to begin REND-019, but
it does not complete REND-018 through REND-020: required validation-clean execution and the
remaining recorded closeout matrix are still outstanding.

The numbered findings below are retained as decision history. Each includes its implemented
resolution; the current gate is missing closeout evidence, not an unresolved architecture defect.

### Correctness Findings And Resolutions

1. **Implement real presentation completion and retirement.**

   `RenderTarget::State::RetirePresentationResources` currently treats the availability of
   swapchain-maintenance or present-ID/present-wait extensions as if it were a completion signal.
   It then decrements a synthetic observation counter after unrelated later presentations. The
   frame path does not attach `VkSwapchainPresentFenceInfoEXT` or `VkPresentIdKHR`, does not call
   `vkWaitForPresentKHR`, and does not implement the promised per-image acquire/present history for
   unextended Vulkan 1.2. Old presentation resources can therefore be destroyed without proof that
   the presentation engine released them, while diagnostics can incorrectly report that explicit
   completion was used.

   Reopen REND-015. Implement and test the actual negotiated extension paths. Revalidate the core
   Vulkan 1.2 strategy against the Vulkan specification. If safe bounded ordinary retirement cannot
   be proved on that floor, use and document a presentation-queue or device-idle fallback rather
   than retaining the current heuristic. Correctness takes precedence over the current no-idle
   aspiration.

   Resolution (2026-07-14): implemented. Maintenance1 now prefers the promoted KHR surface/device
   extension pair and uses the functionally equivalent EXT pair as a compatibility fallback; both
   attach tracked presentation fences as the asynchronous retirement proof. Present IDs are waited
   only before native retirement, with timeout demoted to the successor path. Core Vulkan 1.2
   retains old generations until successor present/reacquire/submission history proves completion;
   counted queue or device idle remains only for terminal paths with no usable successor. Synthetic
   observation counters were removed.

2. **Correct per-frame semaphore and post-acquire failure handling.**

   Render-finished binary semaphores are keyed by frame slot and reused after waiting only for the
   graphics submission fence. A graphics fence does not prove that the presentation queue consumed
   its semaphore wait. Use swapchain-image-indexed present semaphores or a presentation-completion
   primitive that proves safe reuse. In addition, failures after image acquisition can leave an
   acquired image, a signaled binary semaphore, a recording command buffer, or a fence that was
   reset but never submitted. The target is nevertheless left reusable, so a later frame can time
   out indefinitely or violate Vulkan synchronization rules.

   Reopen REND-013 and REND-015. Define the state transition after every fallible post-acquire
   operation, rebuild or retire poisoned frame resources, and add failure-then-next-frame and
   failure-then-shutdown tests rather than checking only the first error code.

   Resolution (2026-07-14): implemented and tested in this change. Render-finished semaphores are
   indexed by acquired swapchain image. Each frame slot also owns a dedicated acquire-completion
   fence. Acquire and submission completion remain independent after a queued submission, and
   retirement waits every fence whose operation is recorded as pending. Any failure after successful acquisition that cannot preserve reusable synchronization
   marks the complete presentation generation poisoned; the next frame retires that generation and
   rebuilds before acquiring again. Deterministic failure-then-next-frame and
   failure-then-shutdown coverage in this change verifies that reset, begin/end, fence-reset,
   submit, and present failures neither reuse poisoned state nor wait forever on work that was
   never queued.

3. **Make swapchain replacement genuinely transactional.**

   Recreation passes the live old swapchain to `vkCreateSwapchainKHR`, then creates image views,
   the render pass, framebuffers, and separate frame resources. Vulkan retires a non-null
   `oldSwapchain` when `vkCreateSwapchainKHR` is invoked even if that invocation fails. A native or
   later dependent-object failure therefore cannot preserve the old target as usable, despite the
   original REND-012 and REND-014 rollback contract.

   Choose a correct commit model: either do not retire the old swapchain until the replacement can
   be committed, or transition explicitly to a recoverable no-swapchain state after the native
   commit boundary. Test failure at every step after `vkCreateSwapchainKHR` and assert the real
   public state, not just destruction call counts.

   Resolution (2026-07-14): implemented and tested in this change. Configuration selection is a
   preflight phase, so its failures preserve the current usable generation. The native creation
   phase reports whether a non-null old swapchain was irreversibly retired, including when
   `vkCreateSwapchainKHR` itself fails; that exact generation becomes ineligible for reuse. Failure
   at or after the boundary leaves `TargetStatus::Active`, recreation pending, and no usable
   swapchain, selected presentation configuration, or reported generation. Retirement bookkeeping is
   reserved before entering that boundary, and incomplete native image enumeration is retried rather
   than published. New native handles are adopted by RAII before later fallible host allocations, so
   allocation failure cannot leak a partial successor or bypass the retirement transition. The next
   acquisition retries with `oldSwapchain == VK_NULL_HANDLE`, while a fully built successor commits
   all owners together and a later retirement-drain error cannot roll it back. Deterministic native-,
   dependent-object-, host-allocation-, retry-, and shutdown-failure tests assert the transition and
   prevent retired handles from being supplied again. The Vulkan rule is documented by
   <https://docs.vulkan.org/refpages/latest/refpages/source/VkSwapchainCreateInfoKHR.html>.

4. **Separate stale swapchains, lost surfaces, and completed recreation.**

   `VK_ERROR_OUT_OF_DATE_KHR` during swapchain work is mapped to `SurfaceLost`, while some genuine
   `VK_ERROR_SURFACE_LOST_KHR` query paths degrade to `BackendFailure`. Acquire or present returns
   `FrameStatus::Recreated` before recreation occurs; the frontend only marks a future recreation
   pending. A resize race can consequently tell the caller to prepare a new surface while
   `RenderTarget::IsSurfaceLost()` remains false and `RecoverSurface()` rejects the replacement.
   Surface recovery also creates recreation metadata whose previous and current revisions are
   equal, which is invalid according to `IsValid(TargetRecreationInfo)`.

   Reopen REND-014 and REND-015. Use distinct typed states for out-of-date/recreation-pending,
   recreation-completed, surface-lost, and device-lost. Never invent a window-state revision for a
   backend-only event, and ensure every reported recovery action is accepted by the corresponding
   public operation.

   Resolution (2026-07-14): implemented and deterministically tested in this change. Target status
   now distinguishes ordinary suspension, `SurfaceLost`, and terminal `DeviceLost` states.
   Out-of-date acquire, present, and replacement results keep the surface valid, return
   `FrameStatus::RecreationPending`, and schedule an internal stale-swapchain retry;
   `FrameStatus::Recreated` is returned only after the replacement commits. Genuine surface loss
   transitions the target to `TargetStatus::SurfaceLost`, returns `RenderErrorCode::SurfaceLost`,
   and is immediately accepted by `RecoverSurface()`, while device loss remains a distinct terminal
   result. Backend-only presentation-change and surface-loss metadata leaves both optional platform
   snapshot revisions absent instead of inventing a revision transition. Tests cover stale acquire,
   stale present, replacement retry, completed recreation, surface recovery, and device-loss
   separation.

5. **Create the promised post-clear, pre-submit intermediate insertion point.**

   `RenderFrame::AcquireFrame` does not acquire a swapchain image or begin recording. `Clear()`
   only stores a color. The monolithic `FinishAndPresent()` later waits, acquires, begins and ends
   the clear render pass, submits, and presents. No generic intermediate layer can append owning
   draw packets after the clear and before final submission, despite that being an explicit
   completion criterion for REND-013 and the readiness premise of REND-020.

   Refactor the frame state machine before the example fixes this API in application code. The
   acquired frame must own real recording state, clear must establish the ordered clear operation,
   a narrow backend-neutral intermediate stage must be able to append work, and finalization must
   submit and present. Add a deterministic ordering test for acquire, clear, intermediate insertion,
   submit, and present.

   Resolution (2026-07-14): implemented and deterministically tested in this change. Acquiring a
   presentable frame now performs image acquisition and begins command recording; `Clear()` records
   the clear immediately and leaves that recording open. One private renderer-owned intermediate
   insertion seam follows the clear, and `FinishAndPresent()` alone finalizes, submits, and presents
   the frame. An ordering test proves acquire, clear, intermediate insertion, submit, and present in
   sequence. This is an internal backend-neutral foundation for a later owning packet contract, not
   a public draw layer, Dear ImGui integration, callback, native-handle escape hatch, or arbitrary
   GPU command API.

6. **Enforce the accepted parent-child and destruction-thread contracts.**

   The resolved LIFE-04/LIFE-05 decisions require parents to outlive children and require
   release-active enforcement. The current bootstrap shutdown merely marks the shared registry as
   shut down; targets and frames retain shared device/target state, so destroying a public parent
   silently extends native resource lifetime. Existing tests explicitly bless shutdown with live
   children. Defaulted destructors also do not verify thread affinity, allowing the last shared
   reference to destroy Vulkan resources on an arbitrary thread.

   Reopen REND-005, REND-011, and REND-016. Prefer an explicit shutdown operation that returns
   `InvalidState` while children are live plus release-active destructor enforcement for ignored
   violations. Add render-thread destruction checks, active-frame teardown cases, move-assignment
   cases, and subprocess/death tests wherever termination is the accepted behavior. If shared-state
   orphaning is intentionally preferred instead, revise every conflicting contract first; do not
   retain both models.

   Resolution (2026-07-14): implemented and tested in this change. The explicit bootstrap shutdown
   operation now returns `InvalidState` without mutation when called from the wrong owner thread or
   while prepared surfaces, devices, targets, or frame tokens remain. Destructors and destination
   move assignments use release-active verification for parent-before-child release and for release
   from the wrong owner/render thread; valid move construction continues to preserve heap-stable
   child state. Every issued frame token, including a suspended, timed-out, or recreation-pending
   token, keeps its target registered until completion or correct-thread abandonment. Focused
   deterministic and subprocess death tests replace the former orphaning behavior and cover
   shutdown retry, transfer-thread teardown, live-child destruction, and move-assignment release.

7. **Guarantee the accepted opaque SDR sRGB output contract.**

   Swapchain selection currently falls back from sRGB formats to UNORM and then to an arbitrary
   first format even though this shaderless milestone has no explicit linear-to-sRGB encoding path.
   Composite alpha can also fall back from opaque to premultiplied, postmultiplied, or inherited
   behavior. That contradicts the accepted opaque SDR sRGB presentation contract.

   For this milestone, require an sRGB format with the accepted nonlinear color space and require
   opaque composite alpha; return `UnsupportedSurface` otherwise. Add deterministic rejection
   tests. Broader format conversion and compositing policy belong in the later 3D/offscreen work.

   Resolution (2026-07-14): implemented and deterministically tested in this change. Swapchain
   preflight now accepts only `VK_FORMAT_B8G8R8A8_SRGB` or `VK_FORMAT_R8G8B8A8_SRGB` paired with
   `VK_COLOR_SPACE_SRGB_NONLINEAR_KHR`, deterministically prefers BGRA when both are present, and
   requires opaque composite alpha. UNORM, undefined, arbitrary/non-sRGB color-space, and
   non-opaque-only surfaces fail with `UnsupportedSurface` before native swapchain creation. The
   selected public configuration must exactly match the native format, color space, alpha mode,
   present mode, and extent. Format and present-mode enumeration retries
   `VK_INCOMPLETE`; adapter compatibility applies the output contract before preference ranking;
   replacement-surface recovery preflights the complete active presentation configuration before
   consuming the handoff; and native creation rejects inconsistent public selected-configuration
   metadata. Linear clear values are encoded by the sRGB attachment write.

8. **Replace process-global Volk dispatch with owner-local dispatch.**

   The production dispatch object wraps Volk's mutable global instance/device function pointers,
   and `volkLoadInstanceOnly`/`volkLoadDevice` overwrite them. The public model permits multiple
   bootstraps or devices but has no process-global exclusivity guard, so concurrent or sequentially
   overlapping owners can use function pointers loaded for another instance or device.

   Reopen REND-006 and REND-010. Store `VolkInstanceTable` and `VolkDeviceTable` in their
   respective RAII owners, or explicitly choose, document, and enforce a one-instance/one-device
   process contract. The accepted architecture currently promises neither a global singleton nor
   that restriction.

   Resolution (2026-07-14): implemented with owner-local Volk dispatch. Loader initialization is
   cached through `std::call_once`, and the small Volk table-population boundary is serialized
   because pinned Volk temporarily uses mutable global loader state while filling tables. Each
   instance and device RAII owner retains its own `VolkInstanceTable` or `VolkDeviceTable` and a
   dispatch derived from that table; downstream device state copies the owning device dispatch.
   VMA receives the owning instance's GIPA/GDPA explicitly. Deterministic fake-dispatch tests keep
   overlapping instance/device owners alive across moves and verify that destruction and allocator
   calls remain bound to the correct owner.

9. **Narrow queue synchronization to the actual queues.**

   One device-wide mutex currently covers a fence wait, image acquisition, command recording,
   submission, and presentation. The bounded waits alone can block another independent target for
   roughly the configured fence and acquisition timeouts, contradicting the narrow shared-queue
   boundary in REND-013 and REND-016.

   Perform target-local waits, acquisition, and command recording outside queue locks. Lock the
   exact Vulkan queue only around externally synchronized queue calls, using the same lock when the
   graphics and presentation handles alias. Add two-target tests that prove one target's bounded
   wait does not stall the other's CPU-side frame preparation.

   Resolution (2026-07-14): implemented with synchronization state owned by each logical-device
   RAII owner and keyed by actual `VkQueue` handle. Fence and presentation-completion waits, image
   acquisition, fence reset, and command recording remain outside queue locks. Queue submission,
   presentation, and queue-idle calls lock only their exact queue; aliased graphics/presentation
   handles share one lock, while distinct queues remain independent. Device-idle and native device
   destruction use an exclusive gate against every queue operation. Deterministic tests cover
   aliased serialization, distinct-queue independence, exclusive device waits, and the production
   finish helper reaching command-buffer finalization and fence reset while another queue user holds
   the shared queue lock.

### Public Contract And API Corrections

1. Replace the Vulkan-flavored public policy names `Fifo`, `PreferMailbox`, and

   preference returns its actual semantic policy and `UnavailableForTarget` fallback. Queued latency
   uses the same strength model and reports the requested maximum, actual maximum, and
   `TargetMaximumExceeded` when a preferred value is reduced. Focused negotiation tests cover exact

3. Make window-state handoff able to request presentation re-evaluation without a pixel-size

   Resolution (2026-07-14): implemented and deterministically tested in this change.
   `RenderTargetSnapshot` now carries `platform::WindowState` and a typed, nonzero
   `PresentationEnvironmentRevision`. The application-side platform-event aggregator owns both
   monotonic per-window revision sequences. Increasing the environment revision schedules and
   coalesces presentation recreation even when pixel size is unchanged; regressing it is rejected.

4. Allow an immediate prepared-surface transfer with the same valid snapshot. The current

   Resolution (2026-07-14): implemented and deterministically tested in this change. Prepared

   ordering validator rejects older snapshots, same-revision conflicts, and presentation-environment
   regressions while accepting a genuinely newer coalesced snapshot. Target updates remain strictly
   newer operations.

   `CreateDevice(RenderDeviceDesc)` always fails. Move fake creation behind a private injection

6. Propagate diagnostics structurally. Native code, symbolic name, and operation are currently

   Resolution (2026-07-14): implemented and deterministically tested in this change. Vulkan result
