# Render Library Open Questions

Status: answered design questionnaire for the first `ponder_render` milestone.

This file now contains only decisions needed to validate Vulkan
create/clear/present/resize/minimize/shutdown across multiple platform windows. Questions whose
implementation belongs to later 3D work are in `open-questions-3d.txt`; Dear ImGui bridge questions
are in `open-questions-ui.txt`.

Recommendations are proposed defaults. The project owner's decision is recorded on each `Answer:`
line, and those answers are the source for `roadmap.txt`.

## Fixed Inputs

These are inputs to the questions, not decisions being reopened here:

- The first implementation uses Vulkan on Windows and Linux. Linux is not yet a verified host.
- Vulkan is not supported on macOS. A later macOS implementation will use Metal directly, not
  Vulkan or MoltenVK.
- The accepted baseline currently names Vulkan 1.2, a dynamic system loader, and private Vulkan
  and operating-system implementation types.
- `ponder_platform` owns windows, native event handling, and native-window discovery. Render owns
  graphics devices, presentation surfaces, GPU resources, and drawing.
- `ponder_math` defines right-handed, Y-up, -Z-forward world space, column-vector transforms,
  column-major matrices, NDC depth `[0, 1]`, and explicit forward-Z/reverse-Z projections. GPU
  packing and backend viewport adaptation belong to render.
- Recoverable failures use `pond::core::Result`; assertions and logging use the core contracts.
- Render must not own chemistry data, project loading, workflow orchestration, UI widgets, or the
  application main loop.
- Public APIs should expose project-owned or standard-library types, not SDL, Vulkan, Metal, or OS
  declarations.

## Contract Reconciliation -- Resolve Before Implementation

- **CON-01:** The conflicting macOS/MoltenVK contracts must be revised or superseded. Should that
  prerequisite update ADR 0007, ADR 0008, the platform boundary, and the platform interop API
  together, or should render record a narrower prerequisite for a separate platform task?

Recommendation: Update ADR 0007, ADR 0008, the platform contract, and render planning together in
one coherent change. Leaving mutually inconsistent accepted contracts would make every later API
decision ambiguous and invite macOS Vulkan assumptions back into the implementation.

Answer: Use the recommendation exactly.

- **CON-02:** Should future presentation compatibility use explicit backend values such as
  `Vulkan` and `Metal`, or should `WindowGraphicsCompatibility` become a backend-neutral capability
  such as `GraphicsPresentable`?

Recommendation: Keep explicit project-owned `Vulkan` and `Metal` compatibility values. Window
creation has genuinely backend-specific native requirements, and naming them is clearer and more
extensible than a vague capability whose meaning changes by platform.

Answer: Use the recommendation exactly.

- **CON-03:** Should backend choice eventually be fixed by the target platform/build, selected at
  runtime, or support both models? This determines whether a parameterless
  `GetRequiredWindowGraphicsCompatibility()` remains sufficient.

Recommendation: Choose a default backend by platform while designing creation APIs to accept an
explicit backend selection later. Make the compatibility query take a backend value so adding
runtime selection does not require replacing a parameterless public contract.

Answer: Backend choice should be controlled by the build. Do not extend the API to allow
selection at runtime. The goal is for the API to remain focused and not add unnecessary features.

- **CON-04:** Until the Metal backend exists, should `ponder_render` on macOS build as a
  backend-neutral library whose renderer/backend creation returns `Unsupported`, be omitted
  entirely, or expose only non-presenting functionality?

Recommendation: Continue building backend-neutral headers and value utilities on macOS, but have
every GPU device/backend creation request return `UnsupportedBackend` until Metal exists. This
keeps consumers portable without implying that headless or presenting GPU functionality works.

Answer: ponder_render on macOS should be configured in CMake to fail to compile. Do not add
unnecessary/temporary features in the API simply to enable building on macOS

- **CON-05:** Should the existing `NativeCocoaWindow`/Metal-layer platform alternative remain as a
  forward-looking Metal contract, be redesigned before Metal work, or be removed until a concrete
  Metal consumer defines it?

Recommendation: Remove its current Vulkan-oriented semantics now, but defer the final Cocoa/Metal
payload shape until the native Metal backend supplies concrete requirements. Prematurely freezing
an unused native contract is more harmful than adding the right closed alternative later.

Answer: Use the recommendation exactly.

## Initial Outcomes And Scope -- Resolve Before Choosing The API Shape

- **SCOPE-01:** What exactly completes the first renderer milestone: validated
  create/clear/present/resize/minimize/shutdown, that foundation plus Dear ImGui, or a first 3D
  visualization?

Recommendation: Make validated create/clear/present/resize/minimize/shutdown the first milestone,
then track Dear ImGui and the first 3D visualization as separate vertical slices. This isolates the
device and presentation contract before shader, scene, and UI concerns can hide lifecycle defects.

Answer: Use the recommendation exactly - scope is only create/clear/present/resize/minimize/shutdown

- **SCOPE-05:** Is rendering to multiple platform windows required initially, required not to be
  precluded, or explicitly deferred despite platform already supporting multiple windows?

Recommendation: Defer multi-window behavior from the first milestone but make presentation targets
independent children of a shared device. That keeps bootstrap small without baking one-window
ownership into the public model.

Answer: Rendering to multiple platform windows is required immediately.

- **SCOPE-10:** What minimum Windows and Linux GPU/driver classes may be required? Must integrated
  GPUs work, or may the initial renderer require a discrete GPU?

Recommendation: Support conformant Vulkan 1.2 integrated and discrete GPUs without vendor-specific
requirements, using capability checks for optional quality paths. Software Vulkan should support
tests and headless automation, not serve as an automatic interactive fallback.

Answer: Use the recommendation exactly

- **SCOPE-11:** What verification is required before the already selected X11 and Wayland paths may
  each be described as supported rather than merely compile-covered?

Recommendation: Require native-host configure, build, live create/present/resize/minimize testing,
and validation-clean shutdown for each path before calling it supported. Compile coverage alone
should be labeled unverified, not treated as runtime evidence.

Answer: Use the recommendation exactly

## Public Architecture And Library Boundary -- Foundational

- **ARCH-03:** Should the public API be backend-neutral from its first revision so the future Metal
  backend consumes the same scene/resource contracts?

Recommendation: Yes. Vulkan and future Metal should implement identical ownership, scene, frame,
capability, and error semantics, with native details confined to implementation and diagnostics so
the first backend does not become the public model.

Answer: Use the recommendation exactly

- **ARCH-04:** Should device lifetime be separate from presentation lifetime, for example one
  `RenderDevice` plus one or more window/offscreen presentation targets, or should a top-level
  renderer be permanently bound to one window?

Recommendation: Separate `RenderDevice` lifetime from target lifetime. One device should own
resources and serve zero or more window/offscreen targets rather than being permanently coupled to
a window.

Answer: Use the recommendation exactly

- **ARCH-06:** Should `ponder_render` be one target with private backend implementations, or should
  backend-neutral frontend and Vulkan/Metal implementations be separate internal targets?

Recommendation: Keep one public `ponder_render` target backed by private frontend/core and Vulkan
implementation targets, with a future private Metal target. This isolates native dependencies and
supports backend conformance testing without exposing implementation targets.

Answer: Use the recommendation exactly

- **ARCH-10:** How should backend-specific capabilities be exposed without either leaking native
  types or reducing every backend to the lowest common denominator?

Recommendation: Expose project-owned capability snapshots with typed features, limits, formats,
and requirement/preference negotiation. Backend identity and opaque numeric diagnostics are useful,
but native handles and structures should remain private.

Answer: Use the recommendation exactly

- **ARCH-11:** What source-compatibility policy should the early public API follow while
  static-first project policy leaves ABI and public value-layout stability explicitly uncommitted?

Recommendation: Mark the API experimental through the foundation and first 3D milestones and allow
documented source-breaking changes during that phase, with no ABI promise. After accepting the
first scene contract, preserve source compatibility within a release series and deprecate changes.

Answer: Use the recommendation exactly - it will be okay to making breaking ABI changes for a
long time. I do not plan on releasing anything permanent for a long time.

- **ARCH-12:** Should public descriptors be freely constructible value types validated by create
  calls, invariant-preserving types with fallible factories, or use both patterns by role?

Recommendation: Use freely constructible value descriptors with safe defaults and transactional
validation at create/update calls. Use strong invariant-preserving types for identities, extents,
formats, and handles where invalid values would otherwise be easy to misuse.

Answer: Use the recommendation exactly


## Lifetime, Threading, And Frame Ownership -- Foundational

- **LIFE-01:** Must renderer creation, presentation, frame submission, resource operations, and
  destruction all occur on the platform owner thread, or may a dedicated render thread own the GPU
  after owner-thread native-handle acquisition?

Recommendation: Make GPU-facing objects affine to a caller-selected render thread, which may be the
platform owner thread or a dedicated application-owned thread. Acquire native window data on the
platform thread, keep device mutation/submission on the render thread, and create no hidden thread.

Answer: Use the recommendation exactly

- **LIFE-02:** If a render thread is allowed, who marshals platform resize/state changes and how is
  shutdown ordered with the owner thread?

Recommendation: The application should marshal coalesced immutable window-state snapshots to the
render thread. Shutdown must stop submissions, drain/cancel render work, destroy targets and the
device, and only then destroy the window and platform runtime.

Answer: Use the recommendation exactly

- **LIFE-04:** Should the top-level device/renderer owner be non-copyable and movable with
  heap-stable state like platform? May it move while child resources or presentation targets exist?

Recommendation: Make it non-copyable and movable with heap-stable state containing a child
registry, so moving it does not invalidate children. Device destruction should perform
release-active verification that the registry is empty; callers must destroy children first, and a
violation terminates rather than risking use-after-free.

Answer: Use the recommendation exactly

- **LIFE-05:** What owns each bootstrap parent-child lifetime: instance bootstrap, device,
  prepared surface, presentation target, frame context, and frame token?

Recommendation: Make the instance bootstrap outlive prepared surfaces and devices; make each device
outlive its targets; let targets own swapchain/frame contexts; and make frame tokens scoped borrows.
Use release-active child registries so invalid teardown order cannot silently create dangling state.

Answer: Use the recommendation exactly

- **LIFE-06:** Should presentation retain a window borrow, store a native snapshot, or require a
  `Window` reference when operations need it?

Recommendation: Consume the borrowed native snapshot synchronously on the platform owner thread to
create a render-owned surface, then retain only that surface and stable diagnostic window identity.
Keep no long-lived `Window` borrow or native snapshot. Require the window to outlive the target and
repeat the owner-thread preparation call when surface recreation needs a fresh native handle.

Answer: Use the recommendation exactly

- **LIFE-07:** What diagnostics can render provide for destroying a platform window before its
  presentation state without changing the accepted caller-enforced lifetime contract?

Recommendation: Attach window identity to target diagnostics, report surface errors with it, and
track live target counts during shutdown. Document that render cannot perfectly observe premature
window destruction rather than implying a guarantee it cannot enforce.

Answer: Use the recommendation exactly

- **LIFE-09:** May shutdown block until the device is idle? Which other public operations, if any,
  may block on GPU completion?

Recommendation: Permit device shutdown, explicit `WaitIdle`, and completion-token waits to block.
Other bootstrap operations should not hide unbounded waits; any acquire or recreation wait must be
documented and preferably bounded or target-local.

Answer: Use the recommendation exactly

- **LIFE-11:** Is the application allowed to build or submit more than one frame concurrently, and
  is frame ordering guaranteed?

Recommendation: Allow multiple frames in flight internally, but only one mutable frame encoder per
target at a time and serialize accepted submissions in a defined order. Callers may prepare
immutable scene/update packets concurrently before ordered submission.

Answer: Use the recommendation exactly


## Device, Capability, Frame, And Presentation Policy -- Foundational

- **FRAME-01:** Should adapter choice be automatic, preference-driven (high performance, low power,
  software), explicitly selectable from public enumeration, or support all three?

Recommendation: Support adapter enumeration, explicit selection by stable enumeration ID, and an
automatic policy with high-performance, low-power, or software preferences. Automatic selection
must satisfy hard requirements first, then apply preferences deterministically.

Answer: Use the recommendation exactly

- **FRAME-02:** Which adapter properties and capabilities must callers be able to inspect before
  choosing a device?

Recommendation: Expose backend/API version, adapter type and identity, driver information,
memory heaps/budgets where available, target presentation support, feature flags, relevant limits,
and supported color/presentation policies. Keep native handles private.

Answer: Use the recommendation exactly

- **FRAME-03:** Beyond the accepted Vulkan 1.2 baseline, what feature floor is mandatory, and which
  features should form optional capability tiers with fallbacks?

Recommendation: Keep the device floor at Vulkan 1.2 plus portable formats/limits needed by accepted
scene features. Require WSI and swapchain support only for a window target; treat descriptor
indexing, advanced synchronization, high MSAA, HDR, and similar enhancements as typed optional
capabilities with explicit fallbacks.

Answer: Use the recommendation exactly

- **FRAME-04:** Is a software Vulkan implementation acceptable as a runtime fallback, test-only
  backend, or never acceptable?

Recommendation: Allow software Vulkan for tests, CI, and explicit diagnostic/headless selection,
but never select it silently when hardware selection fails. An automatic hardware request should
return a clear no-compatible-adapter error.

Answer: Use the recommendation exactly

- **FRAME-05:** Must Vulkan devices with separate graphics and presentation queue families work, or
  may the initial renderer require one family supporting both?

Recommendation: Support separate graphics and presentation queue families from the first backend.
Requiring a combined family would reject valid mainstream devices and leak an avoidable Vulkan
limitation into the product contract.

Answer: Use the recommendation exactly

- **FRAME-06:** Should one device support presentation to multiple surfaces whose support differs,
  and what happens when a later window cannot be presented by the selected adapter?

Recommendation: Let one device serve every surface supported by its adapter. If a later target is
incompatible, return a target-specific unsupported-surface result and let the application choose
another device or rebuild; never migrate resources implicitly.

Answer: Use the recommendation exactly

- **FRAME-07:** Should frame recording use an RAII frame/encoder token that makes invalid call order
  unrepresentable, or explicit `BeginFrame`/`EndFrame`/`Present` calls with state validation?

Recommendation: Return a move-only RAII frame/encoder token with a target-neutral fallible
`Finish`/`Submit`. Window targets then perform or report presentation in their target workflow,
while offscreen targets complete without present; unfinished tokens abandon safely.

Answer: Use the recommendation exactly

- **FRAME-08:** How should a zero-pixel, minimized, hidden, occluded, or otherwise temporarily
  non-renderable target be represented: a successful skipped/suspended status, no frame value, or
  a recoverable error?

Recommendation: Return a successful typed `Suspended` or `Skipped` frame status with no encoder.
These are normal window-lifecycle states, not recoverable failures.

Answer: Use the recommendation exactly

- **FRAME-09:** Should render query the window's current pixel size at recovery boundaries, consume
  explicit application resize notifications, or use both?

Recommendation: Treat platform-provided pixel extent as authoritative through explicit state
snapshots; Wayland has no general independent window-size query. At creation/recovery, validate or
clamp it against Vulkan surface capabilities and refresh native data through the platform owner only
when surface recreation requires it.

Answer: Use the recommendation exactly

- **FRAME-10:** Should out-of-date/suboptimal swapchain recovery be entirely internal, or should a
  frame result report recreation and any required caller action?

Recommendation: Handle ordinary out-of-date/suboptimal recreation internally, while returning
metadata that reports recreation, suspension, or a changed selected configuration. Require caller
action only for surface loss, incompatibility, or lifetime failure.

Answer: Use the recommendation exactly

- **FRAME-11:** Which presentation policies are public: FIFO/vsync, mailbox, immediate, adaptive,
  or a simpler latency policy? What is the default?

Recommendation: Expose semantic `VSync`, `LowLatencyVSync`, and `Uncapped` policies with
`VSync` as default, rather than raw Vulkan modes. Map backend choices internally and report the
actual selected policy.

Answer: Use the recommendation exactly

- **FRAME-12:** If a requested presentation mode, image count, format, or color space is
  unavailable, should creation fail, return the selected fallback, or silently use a documented
  fallback?

Recommendation: Mark descriptor fields as requirements or preferences. Fail unmet requirements;
otherwise return the selected format, color space, latency/image configuration, and fallback reason
instead of silently substituting.

Answer: Use the recommendation exactly

- **FRAME-13:** Is frames-in-flight/latency public configuration or an implementation-selected
  detail? Must the application be able to cap input-to-display latency?

Recommendation: Expose semantic maximum queued-frame latency, defaulting to two frames with a small
documented range such as one through three. Keep raw swapchain image count and synchronization
topology private.

Answer: Use the recommendation exactly

- **FRAME-14:** Is the initial output contract SDR with linear-light rendering into an sRGB
  presentation target? Are HDR and wide-gamut output deliberate extensions rather than accidental
  backend behavior?

Recommendation: Define the initial contract as linear-light rendering with explicit conversion to
an SDR sRGB target. Make HDR and wide-gamut opt-in capability-negotiated extensions with separate
color-space metadata.

Answer: Use the recommendation exactly

- **FRAME-15:** What alpha convention applies to render targets and compositing: opaque-only,
  straight alpha, premultiplied alpha, or selectable by target?

Recommendation: Make ordinary presentation targets opaque and compositable offscreen/overlay
targets premultiplied-alpha, declared explicitly by the target. Avoid straight-alpha intermediate
contracts because they are easier to compose incorrectly.

Answer: Use the recommendation exactly

- **FRAME-17:** On surface loss or device loss, should render recover transparently, expose an
  explicit recreation workflow, or require destruction and reconstruction by the application?

Recommendation: Recover swapchain staleness internally, expose surface loss as target recreation,
and treat device loss as fatal to that device. The application should recreate the device and
repopulate resources; render cannot transparently reconstruct caller-owned state reliably.

Answer: Use the recommendation exactly


## Errors, Diagnostics, And Observability -- Foundational

- **ERR-01:** What numeric range and stable values should `RenderErrorCode` reserve, and which
  caller-actionable distinctions are required?

Recommendation: Reserve `0x0003'0000` through `0x0003'FFFF` for never-renumbered render codes.
Include `InvalidArgument`, `InvalidState`, `UnsupportedBackend`, `LoaderUnavailable`,
`NoCompatibleAdapter`, `UnsupportedCapability`, `UnsupportedSurface`, `OutOfMemory`,
`SurfaceLost`, `DeviceLost`, and `BackendFailure` for the bootstrap milestone.

Answer: Use the recommendation exactly

- **ERR-02:** Which bootstrap outcomes are normal statuses rather than errors: skipped frame,
  timeout, suboptimal presentation, swapchain recreation, and unavailable optional capability?

Recommendation: Model skipped/suspended frames, bounded-wait timeout, suboptimal/recreated
presentation, and unavailable optional capabilities as typed statuses or capability results.
Absence is an error only when the caller marked it required.

Answer: Use the recommendation exactly

- **ERR-03:** Should native `VkResult` values remain diagnostic context only, or be retrievable
  through a deliberately backend-specific diagnostic record?

Recommendation: Keep `VkResult` out of common error codes, but attach an optional project-owned
backend diagnostic with backend kind, numeric code, symbolic text, and validation context. This
preserves detail without exposing Vulkan declarations.

Answer: Use the recommendation exactly

- **ERR-05:** How should Vulkan validation/debug messages map to core log levels and categories, and
  may known noisy messages be filtered or deduplicated?

Recommendation: Route messages through a dedicated render/Vulkan-validation category and map
native severity to core levels. Permit only documented message-ID filters and rate-limited
deduplication; validation tests should fail on unexpected warnings or errors.

Answer: Use the recommendation exactly

- **ERR-06:** Which validation modes should be available and enabled by default in developer builds
  and tests, given that explicitly requested but unavailable validation already fails creation?

Recommendation: Offer off, standard, synchronization, best-practices, and GPU-assisted modes where
available. Auto-enable standard validation in developer builds when present, require standard plus
synchronization in validation jobs, and default release builds off.

Answer: Use the recommendation exactly

- **ERR-07:** Should GPU object names, debug labels, timing markers, and RenderDoc-friendly regions
  exist in every build or only diagnostics configurations?

Recommendation: Compile names, labels, timing markers, and capture regions into every build, then
activate them only when backend support and runtime diagnostics policy permit. Production captures
remain useful without a separate diagnostics binary.

Answer: Use the recommendation exactly










## Dear ImGui And Overlay Composition -- Before The UI Bridge

- **UI-01:** Is Dear ImGui rendering part of the bootstrap milestone, after the first 3D scene, or
  only when the UI library is ready to integrate?

Recommendation: Keep Dear ImGui out of the bootstrap completion criteria, then make it the next
integration milestone before the first interactive 3D scene. This keeps the renderer foundation
independently testable while validating overlay composition early.

Answer: Use the recommendation exactly






## Testing, Reproducibility, And Performance Gates -- Define With The Milestones

- **TEST-02:** Should any Vulkan validation warning or error fail a test? Which message filters, if
  any, are permitted and how are they reviewed?

Recommendation: Fail tests on every validation error and warning by default; informational messages
need not fail. Allow only exact message-ID/vendor/driver-scoped entries with rationale, tracking
issue, owner, and expiry/review condition.

Answer: Use the recommendation exactly

- **TEST-03:** May live GPU/window tests skip when no suitable device, loader, display server, or
  validation layer exists, or is that a failure on supported developer/CI hosts?

Recommendation: Missing capabilities should fail jobs designated as supported render environments;
optional local or compile-coverage jobs may skip with a precise reason. Infrastructure omissions
must not masquerade as passing support.

Answer: Use the recommendation exactly

- **TEST-10:** What Windows compiler/GPU/driver matrix defines initial support? When does Linux move
  from compile-only to a host-tested support claim?

Recommendation: Pin the current project-supported MSVC toolset and current Clang-cl major on
Windows, plus current GCC and Clang majors on a pinned Linux LTS image. Run software-ICD tests every
change and weekly Intel-integrated, AMD, and NVIDIA jobs on pinned production driver branches;
promote Linux only after both compilers and live X11/Wayland pass that matrix continuously.

Answer: Right now, we are only testing on the one Windows laptop I have

- **TEST-11:** Should render adopt PCH-free public-header self-containment tests and compile-only
  consumer/documentation examples from the first public header?

Recommendation: Yes. Add one PCH-free translation unit per public header, an install-tree consumer,
and compile-tested examples with the first API; these prevent accidental transitive includes and
unusable packaging.

Answer: Use the recommendation exactly

- **TEST-13:** Which resource-leak, lifetime-order, synchronization, and validation-clean shutdown
  checks gate each milestone?

Recommendation: Gate every milestone on repeated validation-clean create/destroy/resize cycles,
zero tracked resources at shutdown, correct parent-child diagnostics, and safe retirement of
in-flight resources. Add sanitizers and controlled lifetime permutations where supported.

Answer: Use the recommendation exactly


## Dependencies, Build, And Asset Tooling -- Resolve Before Adding Dependencies

- **BUILD-01:** Are there mandated Vulkan header/function-loading and memory-allocation choices, or
  should the implementation evaluate and pin suitable dependencies under ADR 0001?

Recommendation: Pin Vulkan-Headers, Volk for private function loading through the system Vulkan
loader, and Vulkan Memory Allocator for private allocation under ADR 0001. These focused components
keep native complexity and dependency types behind the backend boundary.

Answer: Use the recommendation exactly

- **BUILD-03:** Should renderer code use the Vulkan C API directly or a private C++ wrapper such as
  `vulkan.hpp`, and what exception/ownership configuration must that choice obey?

Recommendation: Use the Vulkan C API through small project-owned RAII implementation wrappers, with
Volk and VMA private to the backend. This keeps failure propagation, ownership, and destruction
aligned with `Result` without adopting another exception/dispatcher model.

Answer: Use the recommendation exactly

- **BUILD-06:** Which CMake options are required for enabling render, the Vulkan backend,
  validation, and integration tests?

Recommendation: Add `PONDER_BUILD_RENDER=ON`,
`PONDER_RENDER_ENABLE_VULKAN=ON` on Windows/Linux and off elsewhere,
`PONDER_RENDER_ENABLE_VALIDATION=ON` when Vulkan is built, and
`PONDER_BUILD_RENDER_INTEGRATION_TESTS` following `PONDER_BUILD_TESTS`. Select validation at
runtime and reject backend options that conflict with the selected platform; when render is off,
force backend/test options off without probing Vulkan inputs.

Answer: Use the recommendation exactly

- **BUILD-07:** Should configuration succeed without a Vulkan SDK/loader when render is disabled,
  and should a built renderer report a missing runtime loader only at device creation?

Recommendation: Configuration must succeed without Vulkan inputs when render is disabled. With
render enabled, pinned build inputs should suffice; a missing dynamically loaded runtime should
produce an actionable `Result` only when Vulkan device creation is attempted.

Answer: Use the recommendation exactly



## Intentionally Deferred Bootstrap Implementation Details

These private choices belong inside their corresponding roadmap tasks. They are not additional
public design decisions:

- Exact command-pool and command-buffer topology.
- Exact fence, binary/timeline semaphore, and barrier implementation.
- Swapchain format/image-count tie-break algorithms beyond the selected public fallback policy.
- Individual optional Vulkan extensions that do not change the declared capability floor or public
  behavior.
- File and class names for private backend implementation details.
