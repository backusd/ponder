## Dear ImGui And Overlay Composition -- Before The UI Bridge

- **UI-02:** May a dedicated public bridge mention borrowed `ImDrawData` through a forward
  declaration, or must UI translate it into a project-owned draw packet so ImGui remains wholly
  private?

Recommendation: Allow a dedicated opt-in bridge header to forward-declare and borrow `ImDrawData`,
while keeping ImGui absent from the general render API and every backend type private. Translating
ImGui internals into a duplicate packet would add fragile maintenance without strengthening the
important Vulkan/Metal boundary.

Answer:

- **UI-03:** Who owns font-atlas upload/rebuild and its lifetime across device or swapchain
  recreation?

Recommendation: UI should own the ImGui context, atlas contents, and rebuild decision; the render
bridge should own and upload the GPU atlas resource. Preserve it across swapchain recreation, but
rebuild it after device recreation or an atlas revision and retire old resources safely.

Answer:

- **UI-04:** How are application/render textures registered for ImGui and represented as
  `ImTextureID` without exposing Vulkan descriptors or unsafe stale handles?

Recommendation: Register render texture views with the bridge and return opaque,
generation-checked tokens encoded as `ImTextureID`. Registrations survive swapchain recreation,
but device recreation invalidates them and requires explicit re-registration after resources are
rebuilt; native descriptors never escape.

Answer:

- **UI-05:** Is ImGui always the final overlay pass, or must callers control composition order with
  scientific overlays, post-processing, multiple viewports, and offscreen targets?

Recommendation: Make ImGui the final UI overlay by default, represented as a named constrained
composition stage with explicit target selection. Preserve a stable high-level order of scene,
post-processing, scientific overlays, then UI instead of exposing arbitrary backend pass ordering.

Answer:

- **UI-06:** Must the bridge support multiple presentation targets even while secondary ImGui OS
  viewports remain deferred?

Recommendation: Support the bridge across multiple presentation targets from the outset, sharing
device-level font and texture resources while keeping frame state target-local. Secondary
ImGui-created OS viewports can remain deferred without imposing a one-window architecture.
