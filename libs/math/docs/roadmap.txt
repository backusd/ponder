# Math Library Roadmap

Status: implementation plan for the first renderer-ready `ponder_math` library.

This roadmap implements fixed-size, concrete `float` math and collision facilities for 3D
rendering. It does not add renderer resources, backend types, chemistry operations, arbitrary-size
linear algebra, public bulk APIs, or instruction-set-specific public contracts.

The resolved renderer-ready scope includes viewport mapping, project/unproject, and collision. The
projection and NDC decisions still apply. The viewport contract is fixed as continuous coordinates
with a top-left origin and +Y down, NDC edges mapped to viewport edges, no half-pixel offset, and
linear mapping from NDC depth `[0, 1]` to the viewport depth interval. The renderer remains
responsible for configuring Vulkan viewport-Y adaptation.

Each implementation task must leave the repository configured, buildable, and tested for the
behavior it introduces. Every task that adds or changes a public header must add or update that
header's dedicated self-containment translation unit in the same task. The header-test target is
explicitly built without a precompiled header.

## Guiding Decisions

- DirectXMath and DirectXCollision are capability, correctness, and performance references only.
  There is no source, naming, layout, or numerical-compatibility promise.
- DirectX headers may appear only in Windows-only math reference tests. Production library targets,
  public headers, ordinary unit tests, and consumer compile tests never include or depend on them.
- The public vocabulary uses `Vector2`, `Vector3`, `Vector4`, `Matrix3x3`, `Matrix4x4`, and
  `Quaternion` in `pond::math`.
- The initial API contains concrete, unsuffixed `float` types only. Public scalar templates and
  `double`, integer, and half-precision families are deferred.
- Named free functions are preferred. Operators are provided only when their meaning is
  unambiguous. Component multiplication and division, dot products, and cross products are named.
- The world convention is right-handed, Y up, and -Z forward. Values are active column vectors,
  multiplied as `matrix * vector`. `B * A` applies A first and B second. Quaternion composition
  follows the same reading order.
- Matrices use column-major contiguous storage. Translation occupies the final column.
- `Vector2`, `Vector3`, and `Vector4` are compact 8-, 12-, and 16-byte values with public named
  components. `Matrix3x3` and `Matrix4x4` are compact 36- and 64-byte values. `Quaternion` is a
  16-byte value with invariant-preserving accessors.
- CPU layout guarantees do not promise compatibility with vertex, storage-buffer, or uniform-buffer
  layouts. Explicit renderer packing types remain outside math.
- Vectors and matrices default to zero. Quaternions default to identity. Exact `operator==` remains
  distinct from caller-toleranced near comparison and quaternion same-rotation comparison.
- `Radians` and `Degrees` are strong values with explicit conversion. Rotation APIs consume
  `Radians`. Euler construction and extraction are deferred.
- Positions, displacements, directions, and normals use `Vector3`. Named point, vector, and normal
  transform functions encode their different behavior. Dedicated TRS and affine value types are
  deferred.
- Recoverable invalid input and mathematically unavailable results use `pond::core::Result<T>` with
  stable math error codes. Ordinary unchecked scalar arithmetic follows IEEE behavior.
- Default algorithms favor accuracy and documented tolerance stability. Fast-math, reassociation,
  estimate operations, and bit-identical cross-host guarantees are not part of the contract.
- Projection is right-handed and looks down -Z. NDC has +Y up and depth `[0, 1]`. Every projection
  call explicitly selects forward or reverse Z. Finite and infinite-far perspective and finite
  orthographic projections are included.
- Viewport space has a top-left origin and +Y down. Mapping uses continuous edge coordinates, no
  half-pixel correction, and an explicit viewport depth interval.
- The initial collision matrix contains only frustum/AABB classification, frustum/sphere
  classification, ray/AABB interval hits, and ray/triangle hits with barycentrics.
- Collision boundary contact intersects, containment is inclusive, and uncertain culling results
  are retained. Rays and planes normalize valid finite nonzero directions during construction.
- The renderer-ready gate includes scalar, vector, matrix, quaternion, transform, camera,
  projection, clip/NDC, viewport, project/unproject, and the complete initial collision matrix.
- Public bulk APIs, handwritten SIMD, estimate variants, and performance-specific specialization are
  explicitly deferred until representative profiling establishes a need.

## Verification Cadence

Maintain these separate targets:

- `ponder_math_header_tests` contains one translation unit per public header, has no PCH, and has
  only the dependencies inherited by a normal `ponder::math` consumer.
- `ponder_math_tests` contains deterministic, platform-independent unit, property, and regression
  tests. It never includes DirectX headers.
- `ponder_math_reference_tests` exists only on Windows and contains every use of DirectXMath and
  DirectXCollision. It converts conventions explicitly and supplements rather than replaces the
  independent tests.
- `ponder_math_examples` is established in MATH-023 as a PCH-free compile target for the public
  documentation examples and links only `ponder::math`.

### Fast Intermediate Validation

Use the smallest validation set that proves the behavior introduced by the current task:

- Documentation-only changes require no build unless examples, public headers, CMake, or generated
  documentation inputs changed.
- Reconfigure only when CMake inputs, presets, source lists, or target dependencies changed.
- For public API or value-behavior changes, build `ponder_math_header_tests` and
  `ponder_math_tests`, then run the complete `ponder_math_tests` executable.
- On Windows, build and run `ponder_math_reference_tests` only when the task adds or changes
  DirectXMath or DirectXCollision comparison coverage, or when a milestone task asks for it.
- Use independent golden values, algebraic properties, and edge tests on every host. Do not treat
  DirectX behavior as the specification for project-specific failure or uncertainty semantics.
- Keep differential inputs finite, valid, and away from documented collision uncertainty bounds
  unless the test specifically verifies an intentional semantic difference.
- Defer duplicate configurations, Release, sanitizers, clang-tidy, repository-wide formatting,
  whitespace checks, and the portability matrix until the final validation gates.

### Final Validation Gates

At MATH-024 and MATH-025, switch from fast feedback to full completed-library validation:

- Configure clean Debug and Release builds for the primary host.
- Build `ponder_math`, PCH-free public-header tests, unit tests, documentation examples,
  renderer-shaped tests, and Windows reference tests where applicable.
- Run every applicable math test executable or CTest label in Debug and Release.
- Run the supported sanitizer preset, currently Linux Clang ASAN/UBSAN, when that host is available.
- Run clang-format in check mode and clang-tidy over math production and test translation units.
- Run `git diff HEAD --check` and equivalent whitespace checks for intended untracked text files.
- Audit public dependencies, public layout assertions, DirectX isolation, deferred feature absence,
  generated artifacts, logs, build output, and local environment files.
- Run the cross-platform matrix from MATH-025 and record exact OS, compiler, architecture,
  configuration, commands, results, and justified skips.
