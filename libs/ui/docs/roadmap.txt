# UI Rectangle Rendering Foundation Roadmap

Status: implementation plan for Milestone 1 of the project-owned `ponder_ui` stack.

This roadmap implements the rectangle foundation described by
[open-questions.txt](open-questions.txt) and the first entry in
[milestones.txt](milestones.txt). The visible result is intentionally small, but the path is the
same durable boundary intended for later retained UI:

`temporary rectangle facade -> UI paint list -> UI paint compiler -> generic 2D draw packet ->`
`render-owned GPU consumer -> active render frame`

The resolved answers in `open-questions.txt` are binding inputs. UI-000 first verifies that the
renderer foundation is genuinely closed; it must not duplicate renderer fixes that already exist
or hide unfinished renderer work inside `ponder_ui`. The 2026-07-14 source audit reconfirmed the
release-active lifetime checks, owner-local dispatch, exact-queue locking, strict opaque SDR sRGB
selection, real presentation completion, transactional recovery, distinct stale/surface/device-loss
states, and ordered private post-clear seam. The canonical clear-window example is also
implemented. UI-000 nevertheless remains open because REND-018 through REND-020 lack required
validation-clean and complete closeout evidence. This is a verification/provisioning gap, not a
missing renderer architecture or a reason to add a UI workaround.

The renderer-wide shader toolchain, real generic 2D consumer, copied metrics rendezvous, and
rectangle GPU path remain work for this roadmap.

Every production task adds the focused tests needed to protect its behavior. Ordinary tasks perform
only narrow builds, tests, validation, and formatting. UI-016 through UI-021 own the comprehensive
CPU, fake-backend, package, static-analysis, and live Windows/Vulkan gates.

## Guiding Decisions

- Dear ImGui is not part of the architecture, dependency graph, build, tests, examples, or dormant
  compatibility plan.
- `ponder_ui` is the future home of a durable retained-mode scientific desktop UI. This milestone
  defines only its rectangle paint foundation and does not freeze the retained API.
- UI owns logical paint semantics, coordinate and color rules, clipping, and CPU tessellation.
  Render owns generic packet validation, shaders, pipelines, GPU memory, recording, submission,
  presentation, and completion-based retirement.
- Render never depends on UI. The generic `Draw2DPacket` contract is producer-neutral,
  project-private, non-installed, and independently testable by render.
- UI core remains buildable, testable, installable, and consumable when render is disabled. The
  temporary GPU join lives in a separate render-enabled integration component.
- No Vulkan, Metal, SDL, native-window, allocator, descriptor, render-pass, command-buffer, raw
  pointer, callback, or borrowed producer storage crosses the UI/render boundary.
- The paint list contains closed `FillRectangle`, `PushClipRectangle`, and `PopClip` operations in
  painter order. It is owning, immutable after sealing, target-independent, and reusable.
- Geometry is authored in logical units with top-left origin, positive X to the right, and positive
  Y down. Compilation uses exact independent logical-to-pixel extent ratios and preserves
  fractional coordinates.
- Rectangles and clips use finite half-open bounds. Reversed bounds are invalid; zero-area and
  fully transparent fills are successful no-ops after validation.
- Authoring color is bounded sRGB straight alpha. UI converts it to higher-precision linear
  premultiplied color; the first packet stores packed linear-premultiplied RGBA8.
- The packet owns framebuffer-pixel vertices, 32-bit indices, ordered draw records, integer
  scissors, exact pixel extent, and a schema/shader fingerprint. It has no texture, descriptor,
  material, target, native handle, or speculative extension fields.
- One move-only `Draw2DLayer` belongs to one render device and serves compatible targets
  sequentially on the render thread. CPU paint recording and compilation may run elsewhere through
  exclusively owned instances.
- One sealed 2D packet may be recorded after `RenderFrame::Clear()` and before
  `FinishAndPresent()`. It shares the existing render pass, command buffer, submission, and
  presentation path; a second packet is rejected in this milestone.
- All fallible pipeline, validation, allocation, and upload preflight completes before the first 2D
  GPU command. Recoverable failure leaves a clear-only frame that can be finished or abandoned.
- Shader source and semantics are render-owned and portable. Compilation, validation, reflection,
  embedding, and fingerprinting occur offline into the build tree with pinned tools.
- The first Vulkan path stays on the accepted Vulkan 1.2/classic render-pass/sample-count-one
  contract. It uses dynamic viewport/scissor, no depth or culling, and premultiplied-alpha
  blending.
- Upload memory is persistently mapped, bounded, completion-owned, and isolated by target and frame
  slot. Ordinary rendering, resize, and minimization do not use device-wide idle.
- The temporary API is explicitly experimental, move-only, and replaceable. It reports
  `Recorded`, `SkippedSuspended`, or `SkippedEmpty`; submission and presentation remain
  `FinishAndPresent()` responsibilities.
- All milestone testing is Windows-only, per the explicit `UIR-TEST-29` answer. Backend-neutral
  contracts remain portable by design, but no non-Windows compile or runtime support claim follows
  from this milestone.
- Completion requires deterministic, injected-failure, shader, dependency, package, structural
  steady-state, and validation-clean live evidence. An example or screenshot is not an acceptance
  test.

## Scope Exclusions

This roadmap does not implement the retained element tree, stable widget construction API, styles,
themes, layout, text, fonts, images, input, hit testing, focus, controls, scrolling, docking,
accessibility, animation, public offscreen targets, screenshot/readback APIs, a render graph,
arbitrary application shaders, or a general GPU command encoder. It does not add Metal or another
runtime backend.

The first packet has no textures, samplers, descriptors, paths, borders, rounded corners,
gradients, shadows, transforms, non-rectangular clipping, multisampling, or antialiasing. It is not
a public canvas. Deliberate extension points and reconsideration triggers are recorded in
[future.txt](future.txt).

## Verification Cadence

Use one configured supported Windows/MSVC Debug developer tree for ordinary implementation tasks.
Reconfigure only when CMake inputs, dependency wiring, generated-file rules, or source lists change.
Tests accompany the production behavior they protect, but intermediate verification stays focused:

- Format only handwritten C/C++ files touched by the task. Do not perform repository-wide format
  cleanup.
- Build only the directly affected production, header, and focused test targets.
- Run only the exact relevant test executable or named GoogleTest filter. When CTest labels are
  used, pass `--no-tests=error`.
- A changed public header gets a PCH-free self-containment translation unit and focused header build
  in the same task.
- A changed shader contract runs its focused generation, validation, reflection, fingerprint, and
  stale-artifact checks.
- A private Vulkan task uses deterministic fake-dispatch or fake-allocator tests. Run the live WSI
  suite only when the task explicitly requires live behavior.
- Documentation-only tasks run link/terminology searches and `git diff --check`; they do not build
  unless commands, generated contracts, or CMake behavior changed.
- Do not run an all-target build, full CTest suite, Debug/Release matrix, install matrix,
  subsystem-wide clang-tidy, or comprehensive live validation during an ordinary intermediate
  task. UI-016 through UI-021 own those gates.

If a comprehensive gate finds a defect, make the smallest in-scope repair and rerun the focused
test first, then rerun the complete gate. Do not use a late gate to postpone obvious focused tests
from the task that introduced the behavior.

TASK UI-006: Implement The Paint Recorder And Sealed Paint List
--------------------------------------------------------------

Make an owning semantic paint list the canonical CPU input even though the temporary facade emits
only one rectangle.

Work:

- Implement private, type-safe tagged `FillRectangle`, `PushClipRectangle`, and `PopClip`
  commands; do not use a byte stream, raw union, public variant, or virtual hierarchy.
- Store commands and payloads in deterministic owned contiguous storage with stable command-index
  diagnostics and insertion order.
- Model explicit open and sealed states. Mutation after sealing fails, and sealing is
  transactional: failure leaves the recorder valid and reusable.
- Begin with an unpoppable unbounded clip sentinel, reject clip underflow immediately, and reject
  an unbalanced stack at finalization. The compiler will inject the target root clip.
- Validate paint values at the recording boundary, while leaving target-dependent clipping and
  logical-to-pixel work to compilation.
- Enforce checked command, payload, and clip-depth limits before growth.
- Support move-only ownership, capacity-preserving reset, and reuse after successful compilation or
  failed compilation. Add no global recorder or allocator state.
- Permit exclusively owned recorder/list/compiler instances to move across threads without
  promising concurrent mutation.

Done when:

- Focused tests cover ordered mixed commands, invalid fills, zero/transparent fills, nested clips,
  underflow, final imbalance, mutation after seal, every limit boundary, move behavior, reset, and
  deterministic inspection.
- A sealed list is immutable, owning, target-independent, device-independent, and reusable after
  compiler failure.
- No retained-element identity, widget semantics, serialization contract, or public ABI promise is
  introduced.
- Only the UI paint-list target and its focused recorder tests have been built and run.

TASK UI-008: Implement The Deterministic UI Paint Compiler
---------------------------------------------------------

Compile immutable semantic paint into a sealed packet through a pure CPU transaction that is
independent of Vulkan and native windows.

Work:

- Accept only a sealed paint list, immutable matched metrics, and an exclusive reusable packet
  builder/compiler workspace.
- Derive independent X/Y scale from logical and exact framebuffer extents. Reject invalid or zero
  denominators before arithmetic and keep generated positions as top-left framebuffer-pixel
  floats; add no implicit snapping or half-pixel offset.
- Inject a fractional root-target clip, intersect nested fractional clips exactly, clip geometry
  to the current result, and derive conservative integer scissors with floor/ceil and clamping.
- Emit four vertices and six 32-bit indices per visible rectangle with a documented/tested winding,
  triangle-list topology, and painter order.
- Convert straight sRGB authoring color to linear premultiplied color, then quantize to the accepted
  packed packet representation with explicit rounding.
- Elide validated zero-area, fully clipped, and fully transparent fills without disturbing command
  indices or later order.
- Merge only adjacent draw records with identical effective state and scissor when their contiguous
  ranges can be represented safely. Never reorder geometry for batching.
- Check all command, scratch, vertex, index, draw, and byte budgets before growth or writes.
- Make compilation transactional. On failure, the input list and last sealed packet remain valid;
  the exclusive workspace can be reset and reused.
- Expose deterministic private inspection sufficient to diagnose command index, clip depth, bounds,
  output counts, and rejected limit without preserving internal container layout.

Done when:

- Hand-authored tests cover identity and asymmetric scale, fractional coordinates, root/nested
  clips, empty intersections, off-target geometry, scissor rounding/clamping, winding, ordering,
  color vectors, elision, merging, overflow, and every limit boundary.
- Repeated compilation of identical input produces identical semantic packet content.
- Compilation uses no GPU, platform window, render target, backend type, or global mutable state.
- Only the UI compiler and focused packet-validation targets and their exact tests have run; broad
  property/fuzz coverage remains assigned to UI-016.

TASK UI-010: Promote The Intermediate Seam To A Generic 2D Stage
---------------------------------------------------------------

Replace the test-only post-clear marker with one real producer-neutral packet-recording phase while
preserving the renderer's acquired-frame ownership and single submission path.

Work:

- Define a move-only device-scoped `Draw2DLayer` created from one `RenderDevice`. It serves
  compatible targets sequentially, remains render-thread-affine and non-concurrent, and cannot
  outlive its device.
- Select the destination only through an explicit active `RenderFrame`; validate device, target
  state, packet extent, and frame phase without putting target identity in the packet.
- Require clear before 2D recording and recording before finish. Reject draw-before-clear,
  draw-after-finish, wrong device/thread, and a second packet in the same frame.
- Preserve a valid clear-only frame and allow zero packets. Suspended or recreation-pending frames
  skip without consuming the one-stage allowance.
- Record inside the clear-opened render pass and existing command buffer; create no separate pass,
  command buffer, submit, nested frame, or present.
- Represent the order as an internal composition phase that can later grow into explicit content
  and overlay stages without exposing the historical term `intermediate`.
- Require every stage to bind all state it consumes. Expose no callback, native encoder, render
  pass, command buffer, or arbitrary GPU-state escape hatch.
- Add producer-neutral diagnostics for packet, stage, target, and frame-state failures.

Done when:

- Deterministic state tests prove acquire, clear, generic 2D stage, finish, submit, and present
  ordering, plus every invalid transition and clear-only recovery.
- Two targets can use the same draw-layer owner sequentially without ambient target state.
- Render still builds and tests the stage contract without UI.
- Only the affected frame/Draw2D targets and focused state/order tests have been built and run.
