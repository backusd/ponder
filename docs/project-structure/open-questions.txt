PROJECT STRUCTURE OPEN QUESTIONS
================================

Status
------

Closed historical record for the initial project-structure phase.

All questions in this file were answered before implementation. No blocking or
early project-structure questions remain open. The active implementation sources
are now:

    docs/project-structure/decisions.md
    docs/project-structure/roadmap.txt
    docs/project-structure/local-verification.md
    docs/adr/

The body below is preserved only to show what was asked during planning. Do not
interpret question labels such as "blocking" or "early" as current unresolved
work.

Historical Scope
----------------

These questions guided the initial repository and CMake scaffold. The goal was
not to freeze the project forever, but to make the first structure durable enough
that it can support a 10+ year codebase without forcing painful early rewrites.
1. PROJECT IDENTITY AND LONG-TERM CONSTRAINTS
=============================================

Q-ID-001 (blocking): What is the canonical project name?
    Examples: Ponder, Pond, another name. This affects repository names,
    namespaces, executable names, CMake targets, install paths, file extensions,
    documentation, and branding.

Q-ID-002 (blocking): What should the C++ namespace be?
    The current docs mention both `ponder::` and `pond::`. We should choose one
    before creating public headers. Short namespaces are convenient, but the
    namespace should also be unique enough to avoid collisions.

Q-ID-003 (blocking): What should the public include prefix be?
    Examples: `#include <ponder/core/Result.hpp>`,
    `#include <pond/core/Result.hpp>`, or another convention.

Q-ID-004 (blocking): What should the first executable names be?
    Examples: `ponder-desktop`, `ponder-project-inspect`, `pond`, or another
    command naming scheme.

Q-ID-005 (blocking): What should the native project file or directory extension
be?
    Examples: `.ponder`, `.pond`, or a directory with a marker file. This does
    not need to finalize the full project format, but it should align with the
    product name.

Q-ID-006 (blocking): What license should the repository use at launch?
    The high-level roadmap recommends Apache-2.0. Confirm whether that is still
    correct, or whether another license should be used.

Q-ID-007 (early): Should the project require a Developer Certificate of Origin,
Contributor License Agreement, or neither?
    This affects contributor workflow and long-term governance.

Q-ID-008 (early): Should the first repository be a monorepo for platform,
desktop app, built-in plugins, tests, docs, and tools?
    The current roadmap assumes yes. Confirm whether any components should be
    separate from day one.

Q-ID-009 (blocking): What operating systems must be supported during the first
year?
    Suggested initial target: Windows and Linux first, with macOS kept in the
    architecture and CI plan even if not fully supported immediately.

Q-ID-010 (early): What is the minimum supported OS version for each platform?
    This matters for compiler choice, filesystem behavior, packaging, graphics,
    and CI images.

Q-ID-011 (early): Should the initial project optimize for source builds by
developers, or should packaging/installers be considered from the beginning?
    Source builds can come first, but early install/export structure can prevent
    later CMake churn.

Q-ID-012 (blocking): Are future web, remote compute, and headless-worker use
cases considered architectural requirements from day one?
    The high-level roadmap assumes yes, which means no core behavior should live
    only in desktop UI callbacks.


2. CMAKE AND BUILD SYSTEM
=========================

Q-BLD-001 (blocking): What CMake minimum version should be required?
    Choose a version modern enough for presets and good dependency integration,
    but not so new that contributors on common platforms struggle.

Q-BLD-002 (blocking): Should the project start with C++20 or C++23?
    C++23 gives access to newer standard library features, but compiler and
    standard library support must be checked across the supported platforms.

Q-BLD-003 (blocking): What compiler matrix should be supported initially?
    Examples: MSVC and clang-cl on Windows, Clang and GCC on Linux, AppleClang
    on macOS.

Q-BLD-004 (blocking): What are the minimum supported compiler versions?
    This should be chosen together with the C++ standard and dependency manager.

Q-BLD-005 (blocking): Should libraries build as static libraries, shared
libraries, or should this be configurable?
    Static libraries are simpler early. Shared libraries matter for plugin and
    SDK boundaries, but increase ABI and symbol-visibility concerns.

Q-BLD-006 (early): Should plugin interfaces be designed around C++ ABI, a C ABI,
    or a hybrid approach?
    This affects whether shared library boundaries can remain stable across
    compilers and standard library versions.

Q-BLD-007 (blocking): What CMake target naming convention should be used?
    Examples: implementation targets like `ponder_core` with aliases like
    `ponder::core`, or shorter equivalents such as `pond_core` and `pond::core`.

Q-BLD-008 (blocking): What project option prefix should be used?
    Examples: `PONDER_BUILD_TESTS`, `PONDER_ENABLE_WARNINGS_AS_ERRORS`, or
    `POND_BUILD_TESTS`.

Q-BLD-009 (early): Should warnings-as-errors be enabled by default locally, only
    in CI, or behind an option?
    A good default is warnings visible locally and warnings-as-errors in CI once
    the scaffold stabilizes.

Q-BLD-010 (early): Should RTTI be allowed?
    Runtime type information can be useful for editor-like systems and plugins,
    but it should be a conscious choice.

Q-BLD-011 (early): Should exceptions be allowed, banned, or limited by layer?
    This decision affects error handling, plugin APIs, and integration with
    third-party libraries.

Q-BLD-012 (early): Should precompiled headers be used?
    They can improve build times, but may hide include hygiene problems early.

Q-BLD-013 (early): Should unity/jumbo builds be supported?
    They can improve CI and release build times, but should not be required for
    correctness.

Q-BLD-014 (early): Should C++ modules be avoided initially?
    Recommendation: avoid modules at first unless there is a strong reason.
    Tooling and cross-platform support are still uneven for large projects.

Q-BLD-015 (early): Should the build generate installable CMake package exports
    from the beginning?
    This may be useful for plugins and downstream users, but can be deferred if
    it slows the initial scaffold too much.

Q-BLD-016 (early): Where should generated files go?
    Recommendation: generated files should stay in the build tree unless they
    are intentionally checked in and documented.

Q-BLD-017 (early): Should in-source builds be explicitly rejected?
    Recommendation: yes. They make repository state messy and confuse new
    contributors.

Q-BLD-018 (blocking): What should the initial preset names be?
    Examples: `windows-msvc-debug`, `windows-msvc-release`, `linux-clang-debug`,
    `linux-gcc-debug`, `linux-clang-asan`, or shorter names such as `debug` and
    `release`.

Q-BLD-019 (early): Should CMake presets support developer-local overrides?
    Example: `CMakeUserPresets.json` ignored by git.


3. DEPENDENCY MANAGEMENT
========================

Q-DEP-001 (blocking): What dependency management strategy should be used?
    Main options:

    1. vcpkg manifest mode
       Good CMake integration, cross-platform support, version baselines,
       binary caching, and CI reproducibility. Downsides: packages are limited
       to ports available or maintained in vcpkg unless we add overlays, and
       debugging third-party source changes is less direct than vendoring.

    2. FetchContent or CPM.cmake
       Simple for CMake-native libraries and lets us pin exact Git revisions.
       Downsides: dependency builds can become inconsistent, transitive
       dependencies are harder to govern, and non-CMake packages are awkward.

    3. Git submodules plus `add_subdirectory`
       Maximum source visibility and easy local patching. Downsides: larger
       repository checkouts, more responsibility for dependency updates,
       possible license drift, and more manual platform work.

    4. Vendored source snapshots
       Maximum control and offline reliability. Downsides: painful updates,
       repository bloat, and long-term maintenance burden.

    5. System packages
       Simple for some Linux contributors. Downsides: weak reproducibility and
       hard Windows/macOS parity.

    The cost of choosing poorly is high: unreliable CI, hard onboarding,
    dependency version drift, license surprises, and repeated build-system
    rewrites. This is one of the most important early decisions.

Q-DEP-002 (blocking): If vcpkg is chosen, should the project use the public
vcpkg registry only, or also use custom overlay ports?
    Overlay ports would let us pin or patch versions not available upstream.

Q-DEP-003 (blocking): If vcpkg is chosen, should the repository pin a baseline
in `vcpkg.json`?
    Recommendation: yes, so fresh checkouts remain reproducible.

Q-DEP-004 (early): If vcpkg is chosen, should custom triplets be used?
    This may matter for static vs shared builds, runtime library settings, and
    platform-specific graphics dependencies.

Q-DEP-005 (blocking): If submodules or vendoring are chosen, what is the policy
for modifying third-party code?
    Patches should be isolated, documented, and easy to reapply during updates.

Q-DEP-006 (blocking): Which dependencies are allowed in the first scaffold?
    Likely candidates: fmt, spdlog, a test framework, a JSON library, SDL3, and
    Dear ImGui. Rendering and Python dependencies may be deferred.

Q-DEP-007 (blocking): Which JSON or structured data library should be used for
initial configuration and project metadata?
    Options include nlohmann/json, simdjson, glaze, Boost.JSON, or another
    library. The choice affects project-file ergonomics and compile times.

Q-DEP-008 (blocking): Which C++ test framework should be used?
    Options include Catch2, doctest, and GoogleTest.

Q-DEP-009 (early): What license allowlist should dependencies satisfy?
    For a permissive open-source platform, this should be explicit.

Q-DEP-010 (early): Should dependency license information be collected in the
repository from the beginning?
    Recommendation: yes, even if the first version is simple.

Q-DEP-011 (early): Should the project require offline or air-gapped builds?
    This matters for universities, companies, and HPC environments.

Q-DEP-012 (early): Should binary dependency caches be used in CI?
    This can greatly reduce build times, especially with vcpkg.

Q-DEP-013 (early): Should Git LFS be used for large fixtures, sample molecules,
    screenshots, or golden assets?
    Recommendation: avoid large assets at first; decide before adding binary
    fixtures.


4. REPOSITORY LAYOUT AND NAMING
===============================

Q-LAY-001 (blocking): Should the initial layout from `high-level-roadmap.txt` be
implemented exactly, or should it be adjusted first?
    In particular, confirm `apps/`, `libs/`, `plugins/`, `tools/`, `tests/`,
    `cmake/`, `scripts/`, and `docs/`.

Q-LAY-002 (blocking): Should library directory names use hyphens, underscores,
    or single words?
    Example issue: `scientific-data` is readable as a folder name, but CMake
    targets and namespaces may prefer `scientific_data` or simply `data`.

Q-LAY-003 (blocking): Should the scientific-data library be named
`scientific-data`, `data`, `scientific_data`, or something else?
    This is a central concept, so the name should be stable before public
    headers are created.

Q-LAY-004 (blocking): Should every library have public headers under
`include/<project>/<library>/` and implementation files under `src/`?
    Recommendation: yes.

Q-LAY-005 (early): Should private headers live in `src/`, `include-private/`, or
another convention?

Q-LAY-006 (blocking): Should tests live centrally under `tests/`, beside each
library, or both?
    The roadmap currently recommends a central `tests/` directory.

Q-LAY-007 (early): Should each library have its own `README.txt` or design note
from the beginning?
    This may help future contributors understand boundaries.

Q-LAY-008 (early): Should built-in plugins live under `plugins/builtin/`, under
`libs/`, or inside the app until the plugin system exists?

Q-LAY-009 (early): Should small developer tools live under `tools/` even before
they are user-facing?
    Recommendation: yes, if they test or inspect reusable systems.

Q-LAY-010 (early): Should examples live under `examples/` from the beginning, or
wait until the first public API exists?

Q-LAY-011 (early): Should sample project files live under `examples/`,
`tests/fixtures/`, or both?

Q-LAY-012 (blocking): What should be ignored by git?
    Examples: build trees, CMake user presets, IDE files, generated binaries,
    dependency caches, logs, and local environment files.


5. CODE STYLE AND ENGINEERING CONVENTIONS
=========================================

Q-STYLE-001 (blocking): What base formatting style should `.clang-format` use?
    Examples: LLVM-derived, Google-derived, Microsoft-derived, or a custom
    style.

Q-STYLE-002 (blocking): What naming conventions should be used?
    Decide conventions for namespaces, types, functions, member variables,
    constants, files, CMake targets, and options.

Q-STYLE-003 (early): Should filenames use PascalCase, snake_case, or another
style?
    Example: `ProjectFile.hpp` vs `project_file.hpp`.

Q-STYLE-004 (blocking): What line length should be targeted?
    This affects code, docs, generated examples, and review ergonomics.

Q-STYLE-005 (early): Should `.clang-tidy` be strict from the beginning or phased
in gradually?
    Recommendation: start useful but not overwhelming, then tighten in CI over
    time.

Q-STYLE-006 (early): Should include-what-you-use or another include hygiene tool
be part of the workflow?

Q-STYLE-007 (blocking): What should the default error handling style be?
    Examples: exceptions, `std::expected`/expected-like results, status objects,
    or a hybrid by layer.

Q-STYLE-008 (early): What assertion strategy should be used?
    Decide between standard `assert`, custom debug assertions, contract-like
    helpers, or a third-party assertion library.

Q-STYLE-009 (early): What logging style should be used?
    Decide logger naming, categories, severity levels, and whether logs are part
    of the public diagnostics model.

Q-STYLE-010 (early): How should comments and documentation comments be handled?
    Decide when to use ordinary comments, Doxygen comments, or separate design
    docs.

Q-STYLE-011 (early): Should public API headers avoid heavy third-party includes?
    Recommendation: yes, especially for long-term SDK stability.

Q-STYLE-012 (early): Should the project use PIMPL or other ABI-stability
patterns in public SDK-facing types?
    This may matter for plugins, but not every internal library needs it.

Q-CI-002 (blocking): Which platforms should CI build on initially?

Q-CI-003 (early): Should CI run formatting checks before build/test jobs?

Q-DOC-001 (blocking): Should docs be written as `.txt`, `.md`, or a mix?
    The current repository uses `.txt`. Markdown may be more convenient for
    GitHub rendering and public contribution docs.

Q-DOC-002 (early): Should architecture decisions be captured as ADRs?
    Recommendation: yes. A long-lived project benefits from recording why
    choices were made, not only what was chosen.

Q-DOC-003 (early): Should the repository include `CONTRIBUTING`, `CODE_OF_CONDUCT`,
    `SECURITY`, and governance docs from the start?

Q-DOC-004 (early): Should every major library have a boundary document?

Q-DOC-005 (early): Should generated API documentation be configured now?
    Doxygen or Sphinx+Breathe can be useful later, but may be premature before
    stable APIs exist.

Q-DOC-006 (early): Should dependency and license policy be documented before

    Recommendation: yes.

10. DECISIONS THAT ARE PROBABLY SAFE DEFAULTS
=============================================

These can be changed, but they are reasonable defaults unless there is a strong
reason to choose otherwise:

- Use CMake with target-oriented libraries and CMake presets.
- Reject in-source builds.
- Keep scripts as thin wrappers around CMake.
- Use one namespace and include prefix consistently from day one.
- Keep public headers under `include/<project>/<library>/`.
- Keep implementation files under each library's `src/`.
- Keep tests under `tests/` initially, organized by library and test type.
- Use clang-format, clang-tidy, and `.editorconfig` from the start.
- Use dependency version pinning rather than floating latest versions.
- Keep third-party source out of the repository unless a dependency decision
  explicitly says otherwise.
- Avoid C++ modules at the start.
- Avoid large binary fixtures at the start.
- Keep GUI code out of reusable libraries.
