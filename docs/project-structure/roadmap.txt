PROJECT STRUCTURE IMPLEMENTATION ROADMAP
========================================

Purpose
-------

This document breaks step "1. PROJECT STRUCTURE" from
`docs/high-level-roadmap.txt` into implementation tasks that are small enough to
hand to Codex one at a time.

Status: implemented and locally verified on 2026-07-05.

The open questions for this phase have been answered. This roadmap now records
the implemented project-structure scaffold. The project structure should still
remain flexible as we learn, but changes should be made deliberately and
recorded. The guiding principle is that this codebase should look like the
beginning of a project that can survive 10+ years of development.

=========================

Next-Phase Follow-Ups
=====================

The project-structure phase does not block the next roadmap phase. The next
major work should move into project-format and project-loading implementation.

Recommended follow-up tasks for later roadmaps:

- Define the concrete `.ponder` file or directory format under `docs/formats/`.
- Add an ADR if the native project on-disk shape becomes a durable decision.
- Implement project creation, loading, validation, saving, and diagnostics in
  `ponder_project`.
- Make `ponder-project-inspect` load a real `.ponder` project through
  `ponder_project`.
- Add small `.ponder` examples and fixtures once the format exists.
- Install local `clang-format` and `clang-tidy` or document expected versions so
  PS-017 format/static-analysis checks can be rerun.
- Run Linux/macOS configure/build/test and Clang/GCC sanitizer presets when
  those toolchains are available.
- Revisit CI only after local workflows are stable enough to automate.

Decision Baseline
=================

These decisions apply to every task below.

Project identity:

- Canonical project name: `ponder`.
- The C++ namespace is `pond::`. This is the only intended use of the shortened
  name.
- Everything else should use the full `ponder` name: CMake targets, executable
  names, include prefixes, project options, folders, and public documentation.
- Public includes use the form `#include <ponder/{library}/{file}>`, for example
  `#include <ponder/core/Result.hpp>`.
- Native project extension: `.ponder`.
- License: Apache-2.0.
- Contributor certification: Developer Certificate of Origin.
- Initial repository model: monorepo.
- Initial platform support: Windows, Linux, and macOS. Source quality and modern
  implementation practices take priority over supporting old OS versions.
- Initial distribution priority: developer source builds.
- Future web, remote compute, and headless-worker use cases are architectural
  requirements from day one.

Build system:

- Build system: CMake.
- Recommended CMake minimum: 3.28 unless implementation discovers a concrete
  reason to raise it.
- C++ standard: C++23 from day one.
- Initial compiler families: MSVC and clang-cl on Windows, Clang and GCC on
  Linux, and AppleClang on macOS.
- Recommended initial minimum compiler versions: MSVC 19.38 / Visual Studio
  2022 17.8, Clang/clang-cl 17, GCC 13, and AppleClang 15. Raise these if
  needed to preserve clean C++23 implementation.
- Initial library linkage: static libraries.
- Real external plugins should eventually use a stable C-style binary boundary
  with C++ SDK wrappers. Built-in plugins may start simpler.
- Implementation target names use the form `ponder_core`; alias targets use the
  form `ponder::core`.
- Project options always use the `PONDER_*` prefix.
- Warnings-as-errors are always enabled for project code in every build.
- RTTI is enabled intentionally for editor, reflection, and plugin-facing use
  cases.
- Exceptions are allowed only for truly exceptional or unrecoverable cases. All
  project-thrown exceptions should use a single exception type named
  `PonderException`.
- Normal recoverable errors should prefer `std::expected` or an expected-like
  project wrapper.
- Precompiled headers should be used with modern target-scoped CMake practices.
- Unity/jumbo builds are not supported initially.
- C++ modules are not supported and should not be assumed.
- Installable CMake package exports should be prepared for, but not fully
  polished before APIs stabilize.
- Generated files should stay in the build tree unless intentionally checked in
  and documented.
- In-source builds must be explicitly rejected.
- Preset names should be long and descriptive, such as `windows-msvc-debug`,
  `windows-msvc-release`, `linux-clang-debug`, `linux-gcc-debug`, and
  `linux-clang-asan`.
- Developer-local CMake overrides should be supported through ignored
  `CMakeUserPresets.json`.

Dependencies:

- Dependency strategy: Git submodules plus `add_subdirectory`.
- Initial allowed dependencies: spdlog, GoogleTest, nlohmann/json, SDL3, and
  Dear ImGui.
- JSON library: nlohmann/json.
- Test framework: GoogleTest.
- Third-party patches must be isolated, documented, and easy to reapply during
  updates.
- Dependency licenses are permissive by default: MIT, BSD, Apache-2.0, ISC,
  Zlib, and Boost are allowed. MPL and EPL require case-by-case review. Do not
  link GPL or AGPL code into the core. GPL tools may be supported as external
  user-installed engines if they remain separate.
- Dependency license information should be tracked from the beginning.
- Offline and air-gapped builds are best-effort initially. Pin dependencies and
  support pre-populated dependency checkouts, but do not require perfect offline
  builds in the first scaffold.
- CI dependency caches should be used later when CI is added, with careful cache
  keys.
- Avoid large assets at first.

Repository layout:

- The high-level roadmap is guidance, not a rigid file-by-file mandate.
- Directory names for libraries use underscores.
- The generic scientific data library is named `scientific_data`.
- Every library has public headers under `include/ponder/{library}/` and
  implementation files under `src/`.
- Private headers live in `src/`.
- Tests live under a central `tests/` directory.
- Each library has its own `docs/` directory for boundary and design notes.
- Built-in plugins live under `plugins/builtin/`.
- Developer tools live under `tools/` when they test or inspect reusable
  systems.
- Examples live under `examples/` from the beginning.
- Sample project files may live under both `examples/` and `tests/fixtures/`.
- Git should ignore build trees, CMake user presets, IDE files, generated
  binaries, dependency caches, logs, and local environment files.

Code style:

- Formatting: LLVM-derived custom `.clang-format`, 4-space indentation, 100
  columns, Allman braces.
- Naming: types use PascalCase, functions use PascalCase, variables use
  camelCase, private members use `m_*`, constants use `kPascalCase`, CMake
  implementation targets use `project_library`, and alias targets use
  `project::library`.
- Filenames use PascalCase.
- clang-tidy should start useful but not overwhelming, then tighten over time.
- include-what-you-use is not enforced initially. Keep headers clean manually,
  use clang-tidy where helpful, and add IWYU later as an advisory tool.
- Comments should be selective. Use comments for non-obvious logic,
  Doxygen-style comments for important public APIs once stable, and separate
  design docs or ADRs for architectural decisions.
- Public API headers should avoid heavy third-party includes.
- Public SDK-facing types should use PIMPL or other ABI-stability patterns when
  appropriate.
- Assertion macros should be project-defined. They should debug break in debug
  builds and compile out in release builds.
- Logging should support at least trace, debug, info, warning, error, and fatal
  levels. Logging macros should look like `LOG_*`, support `std::format`-style
  formatting, and capture time, file name, and source location.

Testing:

- Test framework: GoogleTest.
- Tests are enabled by default.
- Every scaffolded library should get at least one smoke test.
- Start with one test executable per library. Split into multiple test
  executables as each library grows.
- AddressSanitizer and UndefinedBehaviorSanitizer may be enabled on Clang/GCC.
  Skip sanitizers on Windows for now.
- Code coverage is deferred.
- Benchmark framework setup is deferred until there is meaningful project, IO,
  workflow, rendering, or compute code to measure.
- Golden files and fixtures should be small, readable where possible, and
  intentionally versioned.

Documentation:

- New docs should be written as `.md`.
- Lightweight ADRs should be used for durable decisions such as dependency
  management, plugin ABI, project format, C++ standard, and renderer backend.
- Add `CONTRIBUTING.md`, `SECURITY.md`, and `CODE_OF_CONDUCT.md` before the
  project goes public. Heavier governance docs can wait until there are real
  contributors.
- Every major library should have a boundary document.
- Codex guidance uses a strict two-document lookup model: the root `AGENTS.md`
  for truly global rules, plus exactly one subsystem-local
  `<library|executable path>/docs/Codex.md` file for local guidance.
- Do not add a separate Codex routing index or repo-local skill-pack layer
  unless this decision is explicitly revisited.
- Generated API documentation is deferred.
- Dependency and license policy should be documented before third-party
  dependencies are added.

CI, security, and release posture:

- CI implementation is intentionally deferred right now.
- CMake should still be structured so CI, packaging, install exports, and
  release artifacts can be added later.
- Local scripts should be thin wrappers, not a second build system.
- Dependency vulnerability scanning is deferred with CI.
- Signed releases are a future requirement. Start with signed git tags when
  releases begin; add Windows/macOS code signing before broad binary
  distribution.
- Configure/build should avoid executing plugins, scripts, or external tools
  unless there is a clear reason.
- Aim for reproducible builds, but do not sacrifice more important design goals
  for perfect reproducibility in the initial scaffold.
- If generated code is checked in, the generator version and regeneration
  command must be documented.

TASK PS-003: Create the Root CMake Project
==========================================

Goal:
    Add the root CMake project and establish project-wide build rules.

Depends on:
    PS-001, PS-002.

Implementation notes:
    Add `CMakeLists.txt` at the repository root.

    The root project should:

    - Require CMake 3.28 or newer unless a concrete implementation reason
      requires raising the minimum.
    - Declare the project as `ponder`.
    - Set C++23 as the required standard.
    - Reject in-source builds.
    - Define options with the `PONDER_*` prefix.
    - Enable tests by default.
    - Keep generated files in the build tree.
    - Avoid C++ module setup.
    - Avoid unity/jumbo build setup.
    - Include helper modules from `cmake/`.
    - Add subdirectories only after those subdirectories have meaningful
      `CMakeLists.txt` files.

    The root file should stay small. Put reusable logic in `cmake/` modules.

Done when:
    CMake can configure an empty or near-empty `ponder` project.
    In-source configuration fails with a clear message.

TASK PS-006: Add Dependency Submodule Scaffold
==============================================

Goal:
    Add the selected third-party dependency strategy: Git submodules plus
    `add_subdirectory`.

Depends on:
    PS-002, PS-004, PS-005.

Implementation notes:
    Add submodule locations under `third_party/` for the initial dependency set:

    - spdlog
    - googletest
    - nlohmann_json
    - SDL3
    - imgui

    Pin each submodule to a specific commit or tag. Do not track floating

    dependencies in this task.

    Dependency code is separated from project code.
    No GPL or AGPL code is linked into core project libraries.
