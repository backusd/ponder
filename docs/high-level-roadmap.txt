HIGH-LEVEL ROADMAP
==================

Status
------

This roadmap has been updated after the initial project-structure scaffold was
implemented and verified on 2026-07-05, and after the core foundation
implementation/documentation pass on 2026-07-07. Vulkan renderer and SDL-free
UI integration requirements were frozen by PLAT-002 on 2026-07-08.

The roadmap remains flexible. It should describe the direction of the project,
not freeze every implementation detail forever. When implementation teaches us
that a decision should change, update the relevant roadmap, decision record, or
ADR deliberately.

The first milestone should not be "do chemistry calculations," and it should
not rush directly into user-facing project features. The current foundation
phase solidifies the bottom of the system first: core error handling,
diagnostics, assertions, logging, utility conventions, and cross-cutting
primitives that future libraries will depend on. User-facing milestones should
come after the foundation is boring, tested, and difficult to misuse.


1. PROJECT STRUCTURE
====================

The initial project-structure phase is implemented.

The repository is a CMake-based C++23 monorepo. It is optimized first for
developer source builds and is structured so future CI, packaging, install
exports, web use cases, remote compute, and headless workers can be added later.

Important implemented decisions:

- Canonical project name: `ponder`.
- C++ namespace: `pond::`.
- Public includes: `#include <ponder/{library}/{file}>`.
- Native project extension: `.ponder`.
- License: Apache-2.0.
- Contributor certification: Developer Certificate of Origin.
- Initial dependency strategy: Git submodules plus `add_subdirectory`.
- Initial test framework: GoogleTest.
- Project libraries build as static libraries initially.
- Project code builds with warnings-as-errors.
- CI, packaging, vulnerability scanning, and release artifacts are deferred.

Implemented top-level layout:

Each library has:

- A CMake target named `ponder_<library>`.
- An alias target named `ponder::<library>`.
- Public headers under `include/ponder/{library}/`.
- Implementation files under `src/`.
- A target-scoped private precompiled header.
- A boundary document at `docs/Boundary.md`.
- A subsystem-local Codex guide at `docs/Codex.md`.

Codex guidance follows a strict two-document model:

1. Root `AGENTS.md` for global rules only.
2. The local `<subsystem path>/docs/Codex.md` for subsystem guidance.

Do not add a separate Codex routing tree unless that decision is explicitly
revisited.


2. EXECUTABLES AND LIBRARY BOUNDARIES
=====================================

Initial executable targets are implemented:

ponder-desktop
    The desktop application. It currently starts as a scaffold and should grow
    into the main user-facing application.

ponder-project-inspect
    A developer/user tool for inspecting `.ponder` projects. It currently starts
    as a scaffold and should become the first non-GUI proof that project loading
    works through reusable libraries.

The important architectural rule remains:

    Everything below the application layer must be usable without the desktop UI.

That rule keeps the architecture open for a future web app, command-line tools,
batch execution, cloud services, and remote runners.

Initial library boundaries:

core
    Project-wide foundation code: exceptions, result/error conventions,
    assertions, logging, diagnostics, and small low-level helpers. This library
    should stay at the bottom of the dependency graph.

math
    Vector, matrix, geometry, transforms, bounding volumes, spatial algorithms,
    and numeric helpers that are not chemistry-specific.

chemistry
    Domain objects such as atoms, bonds, molecules, residues, conformers,
    trajectories, basis sets, orbitals, wavefunctions, density grids, and force
    field metadata. This library should describe chemistry data, not know how to
    render it or where computations run.

scientific_data
    Generic scientific data containers: tables, scalar fields, vector fields,
    volumetric grids, time series, images, meshes, annotations, and metadata.

project
    Project loading/saving, schema versioning, asset registry, metadata,
    dependency graph, provenance, cache keys, dirty state, and project
    diagnostics. This is the next major system to make real.

workflow
    Dependency-aware graph model, nodes, ports, typed values, invalidation, and
    execution planning. It should depend on project/data abstractions, not UI.

compute
    Job descriptions, execution backends, scheduling interfaces, local process
    execution, cancellation, progress, logs, artifacts, and remote-runner
    abstractions.

io
    File import/export for chemistry and scientific formats. Start with a small
    set, such as XYZ plus the native project format.

render
    Renderer abstraction, scene data, visualization data flow, GPU-resource
    boundaries, molecular drawing primitives, picking, and screenshot/export
    support.

platform
    Windowing, input, clipboard, monitors, file dialogs, drag-and-drop, and other
    OS integration.

ui
    Editor UI widgets, docking, panels, property inspectors, project browser,
    node editor, plotting, and app-level commands.

plugin_sdk
    Stable public extension contracts for plugins. Real external plugins should
    use a stable C-style binary boundary, with C++ SDK wrappers on top. Built-in
    plugins can start simpler while the API takes shape.

Future executable targets can reuse the same libraries:

ponder-cli
    Command-line tool for inspecting projects, validating project files,
    importing/exporting data, and eventually running workflows headlessly.

ponder-worker
    Headless local or remote compute worker. This should use core, project,
    workflow, compute, chemistry, io, and plugin_sdk, but not ui or platform.

ponder-server
    Future service process for collaboration, remote execution, project
    indexing, or web application support.


3. TEST STRUCTURE
=================

CTest and GoogleTest are implemented.

The scaffold originally started with one test executable per library:

    ponder_core_tests
    ponder_math_tests
    ponder_chemistry_tests
    ponder_scientific_data_tests
    ponder_project_tests
    ponder_workflow_tests
    ponder_compute_tests
    ponder_io_tests
    ponder_platform_tests
    ponder_ui_tests
    ponder_plugin_sdk_tests

REND-002 removed the library-name-only `ponder_render_tests` scaffold. REND-004
adds substantive render value and header tests with the first real public API.

Unit tests live under `tests/unit/{library}/`. Each initial library has a smoke
test proving that its public header can be included and linked.

Future testing should grow in this order:

1. Add focused unit tests as each library gains real behavior.
2. Add project-format tests with small `.ponder` fixtures.
3. Add integration tests across project, IO, workflow, compute, and plugin
   boundaries.
4. Add desktop smoke tests after the UI shell stabilizes.
5. Add performance tests and benchmarks only after there is meaningful workflow,
   IO, rendering, or project-loading behavior to measure.

Fixtures and golden files should stay small, readable where possible, and
intentionally versioned. Large binary assets should remain out of the repository
until a later asset policy explicitly allows them.

CI should eventually run configure/build/test on Windows, Linux, and macOS,
plus formatting, clang-tidy, sanitizers where supported, and packaging smoke
tests once packaging exists. CI itself is intentionally deferred for now.


4. NEXT BUILD SEQUENCE
======================

The project-structure foundation is complete. The core implementation,
documentation, and local verification pass is complete. The next sequence should
bring up a real desktop application foundation early, without hiding platform,
rendering, or UI responsibilities inside the application executable.

The guiding rule for this phase:

    Build the smallest durable foundation that lets the desktop app become real.
    Do not rush project loading, chemistry data, workflow graphs, plugins, or
    compute until the shell, window, renderer, and private UI rendering path are solid.

The desktop application should begin before the `.ponder` project format exists,
but it should not begin with disposable code. The minimal prerequisites for
serious desktop work are:

- `ponder_core` remains the only universal dependency for error handling,
  logging, assertions, UUIDs, build metadata, and low-level utilities.
- `ponder_platform` owns SDL3 window creation, native window handles, event
  pumping, timing, display/high-DPI details, and later platform file dialogs.
- `ponder_render` owns renderer backend selection, graphics context/device
  lifetime, swapchain/surface integration, frame begin/end, resize handling, and
  simple clear/draw behavior.
- `ponder_ui` owns the custom retained UI contract, semantic paint data, and
  CPU tessellation. It submits only owning backend-neutral packets to render.
- `ponder-desktop` owns application orchestration only: startup/shutdown,
  high-level app state, window/main-loop wiring, and user-facing command routing.

That means we can start desktop work almost immediately, but the first desktop
milestone is a durable empty shell rather than a temporary demo.

Step 1: Solidify `ponder_core` (implementation complete)
--------------------------------------------------------

`ponder_core` now provides the dependable bottom of the dependency graph. Before
building project files, workflow graphs, renderers, or chemistry features, keep
the core library's contracts clear and hard to misuse.

The core phase now covers:

- `Result<T>` and `Error` conventions for recoverable failures.
- `PonderException` conventions for exceptional failures.
- Assertion behavior, debug breaks, and release-build behavior.
- Logging macros, severity levels, source locations, and backend boundaries.
- Source-location and stacktrace conventions.
- Build metadata, UUIDs, `ScopeExit`, and minimal UTF string conversion.
- Deferred ownership decisions for cancellation, progress reporting, threading,
  job execution, filesystem access, runtime configuration, and domain typed IDs.

Detailed planning and the final core documentation snapshot live under
`libs/core/docs/`.

Step 2: Verify the completed core phase locally (complete)
---------------------------------------------------------

The final core verification pass should remain the quality bar for future
foundation work: configure and build a supported preset, run `ponder_core_tests`,
run formatting checks, run the scoped clang-tidy pass, and confirm generated
files and local build outputs are not staged.

This step is complete for the current core phase. Repeat the same style of local
verification before treating other foundation libraries as done.

Step 3: Review cross-library dependency direction (documented)
--------------------------------------------------------------

Before adding real desktop behavior, lock in the dependency direction for the
libraries that the desktop shell will touch:

- `ponder_platform` may depend on `ponder_core`, `ponder_io`, and SDL3, but not
  on renderer, UI, project, chemistry, workflow, or compute.
- `ponder_render` publicly depends on `ponder_core` and `ponder_platform`.
  Vulkan/loader/allocator/OS dependencies stay private, as does future Metal
  backend implementation. Its generic 2D contract accepts only project-owned
  packets. Render has no direct SDL target or compile usage and must not depend
  on UI, the desktop app, project, chemistry, workflow, or compute.
- `ponder_ui` owns project paint commands and CPU tessellation and may use
  render's project-private generic 2D contract, but has no direct SDL3 target or compile
  usage. Final executables still receive platform's link-only SDL implementation
  dependency.
- `ponder-desktop` may depend on `core`, `platform`, `render`, and `ui` for the
  first shell milestone. It should not depend on `project`, `chemistry`, `io`,
  `workflow`, `compute`, or `plugin_sdk` until those systems have real APIs.
- `math`, `scientific_data`, and `chemistry` should remain usable without a GUI.

The boundary docs for `platform`, `render`, `ui`, and `apps/desktop` now record
this dependency direction. Revisit those docs before implementation pressure
makes the wrong dependency feel convenient.

Step 4: Build the platform window and event foundation
------------------------------------------------------

Implement only the platform behavior needed to host a real desktop window:

- SDL3 initialization and shutdown with RAII ownership.
- Window creation/destruction with title, size, visibility, and high-DPI data.
- Event polling with typed events for close, resize, focus, keyboard, mouse, and
  frame timing.
- Vulkan-compatible native data only through the closed Win32, X11, and Wayland
  platform variant selected by ADR 0008. Native Metal interop is deferred until
  the macOS backend is designed.
- Core logging and `Result`-based failure reporting.

Keep this app-neutral. No project loading, menus, molecule data, rendering
policy, or ImGui logic belongs in `ponder_platform`.

Step 5: Build the renderer bootstrap foundation
-----------------------------------------------

Implement enough `ponder_render` to prove that the application can render to a
platform window through a stable renderer boundary:

- Implement the Vulkan backend selected by ADR 0008 for Windows and Linux.
  Establish the approved Vulkan loader/header and allocator dependencies before
  renderer code. macOS render support waits for a native Metal backend.
- Call `pond::render::GetRequiredWindowGraphicsCompatibility()` before window
  creation and create renderer-owned surfaces from the platform native-window
  variant.
- Own graphics device/context lifetime, swapchain or surface lifetime, frame
  begin/end, resize handling, vsync policy, and a clear-color pass.
- Expose a small renderer API that the desktop app can drive without knowing the
  backend details.
- Keep renderer code independent from project files, chemistry models, and UI
  widgets or paint semantics.

The goal is not a molecule viewport yet. The goal is a non-temporary render loop
that can clear the window every frame and survive resize/shutdown cleanly.

Step 6: Build the project-owned rectangle rendering foundation
--------------------------------------------------------------

Implement enough `ponder_ui` and the generic render 2D layer to draw one rectangle
through the production-shaped path described by ADR 0010. Do not design the
public retained widget system in this step.

- UI records semantic paint commands in logical coordinates and compiles them
  into owning pixel-space packets after exact frame metrics are known.
- Render consumes only generic indexed 2D geometry and owns shaders, pipelines,
  GPU memory, command recording, submission, and completion retirement.
- Establish deterministic ordering, rectangular clipping, linear premultiplied
  color, resize/suspension behavior, and multi-target-safe frame lifetimes.
- Add independent paint/compiler tests, generic packet/backend tests, shader
  interface checks, failure injection, and a bounded live rectangle workbench.
- Keep the temporary rectangle entry point explicitly experimental; text, layout,
  retained widgets, input, styling, and the stable public API are later milestones.

Step 7: Assemble the first durable `ponder-desktop` shell
---------------------------------------------------------

Grow `ponder-desktop` from scaffold to a real application shell using only
`core`, `platform`, `render`, and `ui`:

Begin this step only after the retained UI public contract is separately designed. Desktop consumes
that facade and never calls the temporary rectangle harness or generic packet API directly.

- Startup/shutdown logging and build metadata display.
- Window creation, main loop, event dispatch, frame timing, and clean exit.
- Obtain the selected renderer's compatibility requirement before creating the
  platform window rather than hardcoding a backend flag in the application.
- Renderer clear pass every frame.
- Retained UI dockspace/menu/status/log shell.
- Basic command routing for commands that do not require project state yet, such
  as Quit, About, toggling panels, and showing diagnostics.

This should be the first user-visible milestone: a desktop app that opens a
window, renders every frame, hosts the retained UI shell, and has clean subsystem boundaries.

Step 8: Verify the desktop foundation locally
---------------------------------------------

Before moving into project/data work, verify the empty shell as a real product
foundation:

- Configure and build the normal developer preset.
- Run unit tests for `core`, `platform`, `render`, and `ui` as those tests exist.
- Run formatting and scoped clang-tidy for touched libraries.
- Launch `ponder-desktop` locally and confirm it opens a window, renders a clear
  frame, displays the retained UI, handles resize, and shuts down cleanly.
- Confirm no generated files, logs, build outputs, or local environment files are
  staged.

Only after this shell is trustworthy should project loading become the main
thread of work.

Step 9: Add foundational math primitives needed by the shell and viewport
------------------------------------------------------------------------

Add only the `ponder_math` types needed broadly by renderer, viewport, and later
chemistry systems: vectors, matrices, transforms, bounds, camera helpers, numeric
helpers, and geometry primitives if they are truly shared.

Do not build renderer-specific draw code or chemistry-specific molecular logic
in `ponder_math`. Keep the library useful without SDL, ImGui, project files, or
chemistry data.

Step 10: Add foundational scientific and chemistry data models
--------------------------------------------------------------

Add minimal value types and containers needed to describe simple scientific and
chemical data:

- Elements, atoms, bonds, molecules, conformers, and simple metadata.
- Domain-specific typed IDs in the owning domain library, not in core.
- Tables, arrays, grids, or fields only when a near-term use case needs them.

These models should be useful without project files, rendering, IO, workflow, or
compute.

Step 11: Define the native `.ponder` project format
---------------------------------------------------

After the desktop shell exists and the first data model boundaries are clear,
define the native project format under `docs/formats/` and add an ADR if the
file vs directory shape needs a durable decision.

The extension is already decided: `.ponder`. A directory-based project remains a
good candidate:

    Example.ponder/
        project.json
        assets/
        cache/
        logs/

Keep the first schema small, explicit, versioned, and aligned with the core
error/diagnostic model.

Step 12: Implement project loading and validation
-------------------------------------------------

Build real behavior in `ponder_project` for project creation, loading,
validation, saving, and diagnostics. Use core `Result`, `Error`, logging, and
assertion conventions consistently.

Keep this library independent from the desktop app. The desktop app should be
one consumer of the project API, not the place where project semantics live.

Step 13: Wire `ponder-project-inspect`
--------------------------------------

Make `ponder-project-inspect` load a `.ponder` project through the project
library and print a concise summary of schema version, metadata, assets, and
diagnostics.

This gives us a GUI-free smoke test for the same project APIs the desktop app
will use.

Step 14: Add initial IO support
-------------------------------

Add a small `ponder_io` handler interface and one simple molecule importer,
likely XYZ. Use small fixtures under `tests/fixtures/`.

Keep importers independent from desktop UI. Imported data should flow through
`ponder_io` into `ponder_chemistry` or `ponder_scientific_data` models, and only
then into project or viewport systems.

Step 15: Wire project loading into the desktop app
--------------------------------------------------

Add Open Project behavior through the platform/UI command system and route
loading through `ponder_project`. Show diagnostics and populate the project
browser with metadata and assets.

At this point the application should be able to load a real `.ponder` project,
but the shell should already have existed without project-specific temporary
code.

Step 16: Add a minimal viewport and molecule preview
----------------------------------------------------

Add enough render/platform/ui integration to show a simple molecule preview for
loaded molecule assets. This proves the early data path:

    project -> asset registry -> molecule data -> scene data -> viewport

Keep molecule preview code behind renderer/scene boundaries rather than embedding
chemistry-specific drawing logic directly in the desktop app.

Step 17: Introduce plugin extension points
------------------------------------------

Add the first `ponder_plugin_sdk` interfaces only after the built-in project and
IO path works. Start with file-format handlers and simple metadata.

Step 18: Introduce workflow graph primitives
--------------------------------------------

Add a minimal workflow graph with nodes, ports, typed values, dependencies, and
invalidation.

Step 19: Introduce compute job abstractions
-------------------------------------------

Add job descriptions, progress, cancellation, logs, artifacts, and a local
runner interface. Do not integrate a large external chemistry engine until the
job model is clear.

Step 20: Expand toward real chemistry integrations
--------------------------------------------------

Once project loading, asset handling, plugin registration, workflow graph, and
compute jobs exist, begin integrating external engines behind adapters. Start
with one engine and one narrow workflow.

5. NEXT MILESTONES
==================

The current substantial milestone is a foundation milestone, not a user-facing
feature milestone:

    `ponder_core` is stable enough that other libraries can rely on its error,
    diagnostic, assertion, logging, and low-level utility conventions without
    needing to revisit those contracts immediately.

That milestone is now represented by the completed core implementation,
documentation, and local verification pass. It proves:

- Core systems are reusable outside the GUI.
- Recoverable errors and exceptional failures have clear conventions.
- Assertions and logging are testable and consistent.
- The bottom of the dependency graph is small, boring, and documented.
- Future project, workflow, plugin, rendering, and compute systems have a stable
  foundation.

The next user-facing application milestone should now be smaller and earlier:

    A desktop application that opens a native window, renders every frame,
    displays a durable retained UI shell, handles resize/shutdown cleanly, and
    keeps platform, renderer, UI, and application responsibilities separated.

That milestone proves the most important early desktop architecture claims:

- The app is desktop-first but not desktop-only.
- Platform/window code is reusable outside the app executable.
- Renderer setup is not hidden in application code.
- UI paint semantics stay in the UI library and are not scattered through the desktop app or
  renderer.
- Future project loading can attach to a real shell instead of creating the
  shell and project features at the same time.

The following user-facing milestone should then be:

    A desktop application that opens a `.ponder` project, validates project
    metadata, lists project metadata and assets, imports a simple molecule file
    into the project, saves the updated project, and shows a basic molecule
    preview.

That later milestone proves the project format, data model, IO, project library,
and viewport path without delaying the first real desktop foundation.
